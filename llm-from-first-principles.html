MENTAL MODEL / AI SYSTEMS

LLM from First Principles

Search、Translation、Verificationとして捉え直し、TrainingからInference、Hardware、causality、研究動向、Codex利用までを一つの流れで理解する。

公開情報の基準日: 2026-07-13 ・ 一次論文と公式資料に基づく

主張する結論

LLMは、weights、context、external sourceからpatternを検索し、prompt、specification、code、image、Tool resultを別の表現へ翻訳するsystemである。truth、causality、responsibilityは自動では付かないため、Verifier、Permission、Human Approvalを外側に置く。

1. 結論を先に置く

SearchはWeb Searchだけではない。Translationもlanguage-to-languageだけではない。この二つを広く、しかし曖昧にならないよう定義する。

Parametric Search
weightsに圧縮されたpatternから、current contextに合うcontinuationを活性化する。
Context Search
Attentionがprompt、conversation、retrieved document、Tool resultの関連位置を選ぶ。
External Search
Web、repository、database、file system、runtimeをToolで観測する。
Translation
ある表現を別の表現へ写す。question-to-answer、spec-to-code、code-to-test、error-to-patch、plan-to-Tool callを含む。

重要な限定: これはLLMが内部で候補を総当たりするというliteral claimではない。Transformer内部にはdistributed representationと大量のmatrix computationがある。このMental Modelは、何を得意とし、何を別途保証しなければならないかを示す。

この結論の根拠

  1. Pre-trainingは主にnext-Token conditional probabilityを最適化し、truthやcausal effect自体をlossにしていない。
  2. Attentionはcontextから情報を選び、MLPは表現を変換する。
  3. Toolは外部観測とactionを増やすが、Modelへ自動でresponsibilityを与えない。
  4. formal causal benchmarkでは改善例と同時に、temporal orderをcausationと誤認するfallacyも確認されている。

2. Training: 能力が作られるphase

「LLMをTrainingする」は単一工程ではない。何を学習可能にするか、広いpatternをどう圧縮するか、どのbehaviorを選びやすくするかは別phaseで決まる。

Pre-trainingの一回のweight update

  1. Forward Pass: Modelが各positionでvocabulary全体のLogitを計算する。
  2. Softmax: Logitを合計1のprobability distributionへ変換する。
  3. Cross-Entropy: 正解Tokenへ割り当てたprobabilityが低いほど大きなlossにする。
  4. Backpropagation: lossへの寄与を各weightへ伝える。
  5. Optimizer: lossが下がる方向へweightを少し更新する。

多数GPUではData Parallel、Tensor Parallel、Pipeline Parallel、Expert Parallelを組み合わせる。AllReduceは複数deviceのgradientを集約するcommunication operationである。FP8やBF16はmemoryとcomputeを節約するlower-precision formatだが、numerical stabilityの管理が必要になる。

3. Inference: Requestからresponseまで

Inferenceは一回のModel callではない。Application、Orchestrator、Model Server、Runtime、Acceleratorが連なり、Tool callがあると外部loopへ出て再びModelへ戻る。

4. Causality: 構造上の弱さを正確に言う

LLMがcausal languageを生成できることと、causal relationをdataから同定できることは別である。

通常のnext-Token Pre-trainingが直接最適化するのはobserved sequenceのlikelihoodである。Training textに「Aの後にB」「AとBが一緒に語られる」というcorrelationがあっても、confounder、reverse causation、selection biasを除くexperimentがあるとは限らない。

限定された主張: Transformerがcausal representationを絶対に持てないのではない。observational next-Token objectiveだけでは、causal graph、Intervention、Counterfactualを一意に正しく学ぶことが強制されない。

論文から分かること

  • Causal Parrotsは、causal factを語る成功がTraining text内のcausal fact同士のcorrelationで説明できる可能性を論じた。著者のconjectureを含む。
  • CLadderはAssociation、Intervention、Counterfactualを含むformal causal questionがLLMに難しい一方、CausalCoTで改善することを示した。
  • Joshi et al. (EMNLP 2024)はmention orderやtemporal orderからcausationを誤推定するpost hoc fallacyを報告した。
  • Kasetty et al.はcontrolled Intervention benchmarkでGPT系Modelに有望なperformanceも報告した。

結論は「LLMはcausal reasoning不可能」ではない。explicit causal structure、experiment、Tool、Verifierを与えれば改善できるが、native guaranteeはない。

Codexでの読み替え: 「このchangeでbugが直る」はhypothesisである。failing Testを再現し、single changeをInterventionとして実行し、failureが消えるか再測定することでcausal evidenceへ近づく。

6. LLMに対する誤解

「次Token予測だけだから単純」

objectiveが短く書けても、内部に抽象feature、algorithm-like circuit、planning behaviorが生じうる。mechanismの短さはfunctionの単純さを意味しない。

「SearchとTranslationだけなら新しいものを作れない」

learned representationをconstraintの下で再構成すれば、新しいprogramやdesignが生じうる。Noveltyとcorrectnessは別である。

「Chain of Thoughtは内部思考の完全な記録」

visible reasoningはoutput channelであり、読みやすくTrainingされうる。internal activationの完全なtranscriptではない。

「Toolを使えばhallucinationは消える」

wrong query、bad source、prompt injection、misread result、Tool side effectが残る。RetrievalとVerificationは別工程である。

「low Temperatureなら正しい」

同じhigh-probability outputを選びやすくするだけで、weightsやcontextにある誤りは直らない。

「long contextなら全て覚えて使える」

Context Windowへ入ることと、必要箇所を正しく利用することは違う。distractor、position、retrieval quality、KV Cache costが影響する。

「causal explanationが自然ならcauseを理解している」

causal languageのretrievalかもしれない。Intervention、Counterfactual、alternative hypothesisのTestが必要である。

7. Codex等を使う人の実践ルール

Coding Agentへ任せるのはSearchとTranslationである。正しさを決めるevidence、actionを許すpermission、結果へ責任を持つapprovalは外側へ残す。

  1. Scopeを狭くする: repository、directory、file、non-goalを指定する。
  2. Success Criteriaを書く: Test名、expected behavior、performance threshold、browser state等を観測可能にする。
  3. Source of Truthを指定する: spec、issue、schema、official documentation、existing Testの優先順位を決める。
  4. Search結果をuntrusted inputとして扱う: Web pageやrepository textには古いinstructionやprompt injectionがありうる。
  5. Translationをdiffで確認する: explanationではなくfile diff、command、dependency、artifactを見る。
  6. Causal claimをTestする: 再現、single change、再測定でhypothesisを検証する。
  7. Permissionを分離する: network、secret、production、delete、payment、publishは必要時だけapproveする。

Prompt例: 「login bugを直して」ではなく、「auth moduleで再現するfailing Testを先に追加し、public APIを変えず最小patchを作り、関連Testとfull suiteの結果、diff、残るriskを報告する。production credentialとnetworkは使用しない」と依頼する。

OpenAIのCodex sandbox資料はnetworkをdefaultで制限し、workspace外writeを制限し、unsandboxed actionへapprovalを要求する設計を説明している。Sandboxはcorrectness guaranteeではない。誤ったactionのreachを狭めるboundaryである。

8. 最終Mental Model

LLMは、三種類のSearchと多段のTranslationを行うprobabilistic engineである。Parametric Searchはweightsからpatternを取り出し、Context SearchはAttentionでcurrent inputの関連箇所を選び、External SearchはToolでworldを観測する。TransformerはそれらをTokenやTool callへ翻訳し続ける。しかしobservational Trainingはcausality、truth、responsibilityを直接保証しない。Toolを持つLLMを使うときは、Verifierで結果を確かめ、Permissionでactionを制限し、Humanが最終責任を持つ。

Useful AI System = Search + Translation + Verification + Permission + Human Responsibility

SearchとTranslationはLLMの強みである。VerificationとPermissionを省くと、その強さがそのままriskになる。

このMental Modelの限界

  • LLM内部の全algorithmをliteralに記述するものではない。
  • LLMがcausal taskを一切解けないとは主張しない。explicit structure、fine-tuning、experimentで改善できる。
  • consciousnessやmoral statusは扱わない。functional capabilityとは別の問いである。
  • closed frontier ModelのTraining data、exact architecture、internal circuitには非公開部分がある。
