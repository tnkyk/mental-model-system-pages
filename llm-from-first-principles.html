LLM from First Principles

Mental Model Studio · 2026-07-15 · 投資・製品ランキング予言ではない

最終メンタルモデル

LLMは、重み・文脈・外部情報から次の出力に役立つパターンを探索し、質問・仕様・コード・Tool結果などの表現を別の表現へ変換する仕組みである。学習は主に観測系列の予測であり、真偽・因果介入・責任を直接保証しない。したがって実用では探索と変換を機械に担わせ、検証・権限・人間の承認を外側に置く。

有用なAIシステム = 探索 + 変換 + 検証 + 権限 + 人間の責任

すでに触れている裂け目（Lived stake）

一つ目の場面。社内のナレッジチャットに「このAPIの現行の認証方式は何か」と聞く。答えは滑らかで、自信ありげで、用語も正しい。三日後、実装を見たらそのAPIは既に別方式へ移行していた。画面上の流暢さは、正しさの代理変数ではなかった。あなたは「嘘をつく主体」に騙されたのではなく、観測系列から次にありそうな続きを出したプログラムのふるまいに、信用を渡しすぎた。

二つ目の場面。モデルが長い手順を「まずは〜を確認し、次に〜を検討し、結論として〜」と書き連ねる。文章は思考の跡に見える。レビュー会で誰かが「Chain of Thoughtがあるから大丈夫」と言い、検証を後回しにした。見える推論文は出力チャネルの一つであり、内部活性化や因果計算の完全な転写ではない。もっともらしい手順書は、正しさの証明書ではない。

三つ目の場面。Coding Agentが差分を提案する。「このロックは冗長です。削除します」と説明し、自信を持ってパッチを出す。そのロックは稀なレース条件を防いでいた。Agentは探索と変換は得意でも、本番への到達範囲と承認の境界は自動では与えられない。危険な変更提案は、モデルが「悪い」からではなく、権限レイヤが流暢さに引きずられたときに起きる。

これらは「AIが賢いかどうか」の話ではない。あなたがすでに感じている流暢さ・説明・自信と、正しさ・因果・権限の裂け目である。

この記事がすること／しないこと

すること：上の三つの裂け目を、再構築可能な機構（目的関数・層・サンプリング・配信・Tool・検証・権限）へ分解する。Coverage Map の順に、なぜ「保証なき候補生成」が工学的に選ばれ、どこで検証と権限が外側に残るかを鎖で説明する。Attention / Sampling / Evidence の各 Lab（いずれも教育用トイ）と接続し、途中で Everyday phenomena（P1–P6）を回収する。

しないこと：価格・投資・採用ROI・どの製品が一位かの予言。プロンプト技のカタログや、閉鎖frontierの非公開回路を事実扱いすること。意識・道徳的地位の結論。RAGや長期記憶製品の完全仕様。

読み終えたとき、あなたは「このモデルは信じられるか」ではなく、「自分は今、探索・変換・検証・権限のどの層に立ち、何を機械に任せ、何を外側に残しているか」を説明できるはずである。

探索の三層（モデル補足）

  • 重み内探索: 重みに圧縮されたパターンから、現在の文脈に合う続きを活性化する（parametric）
  • 文脈内探索: Attention がプロンプト・履歴・取得文書・Tool結果の関連位置を選ぶ（context）
  • 外部探索: Web・リポジトリ・DB・ファイル・実行環境を Tool で観測する
  • 変換: ある表現を別の表現へ写す（質問→回答、仕様→コード、ログ→修正案、計画→Tool呼び出し）

1. N1 — Lived stake: 流暢さと信頼の裂け目

（冒頭の三場面が N1 本体である。）

読者が最初に固定すべき契約は一つだけである。流暢な文章・自信ありげな口調・長い手順書・「私は確かだ」というメタ発言は、いずれも信用や権限の代理変数ではない。それらは生成されたトークン系列の見た目である。この記事の以降の節は、なぜそうなるかを機構から再構築し、どこから検証と権限が外側に残るかを示す。N1 は「用語の定義」ではなく、あなたがすでに損をしうる裂け目の命名である。

次の限界

裂け目を感じたあとで最初に落ちる罠は、「ではLLMはダメだ／万能だ」の二択である。次節は、なぜ観測系列の予測という目的が工学的に選ばれ、その目的が何を保証しないかを先に置く。

2. N2 — 観測的予測のトレードオフ：なぜ「保証なき候補生成」が選ばれるか

前層の限界 / why_exists

N1 の裂け目だけを見ると、「正しさを直接最適化すればよいのに」と感じる。だが実務が直面する制約は別である。世界の真偽ラベル付きコーパスを、あらゆる質問とあらゆる未来状態について用意することはできない。その一方で、人間が書いたテキスト・コード・対話の山は、すでに大量に存在する。工学は「真理関数を直接学習する神託」ではなく、「観測された系列の次を当てる」方向へ進んだ——それが主目的を観測尤度に置く理由である。

仕組み

一般的な事前学習の目的は、観測系列 x₁,…,x_T に対して、条件付き確率 P(x_{t+1}|x_{≤t}) を高くすることである。これは連想（association）の言語に近い。テキスト上で「Aの近くにBが来る」「エラーメッセージの次に修正案の型が来る」といった統計を、巨大な関数近似器に圧縮する。真偽判定器や因果同定器を直接損失にしていないため、流暢な誤答・表面相関の再生産・因果語彙の模倣が同じ最適化の裏で起きうる(*9)。

ここで重要なのは、「だから役に立たない」ではない。観測尤度の最適化は、候補生成の広大な空間を一度に開く。自然言語・コード・表・Tool呼び出し形式など、同じ「次トークン」の器に載る表現を多種学べる。その代償として、正しさ・介入効果・実行許可は自動では同定されない——それが N2 の hard claim である。

Association と truth を混ぜると P6 が生まれる。「ブロックチェーンだから改ざん不可」に似た絶対語（「学習済みだから正しい」「推論したから正しい」）で LLM を語ると、観測目的が保証する範囲を超える。

ここでP6が回収され始める: 絶対語は性質の誤読である。観測尤度の最適化は、改ざん耐性や真理保証のアルゴリズムではない。

worked example

社内Wikiに古い認証方式の説明が残っている世界を考える。事前学習や検索でその文章を多く見れば、モデルは現行の正しい方式より、古い方式の言い回しを流暢に再生産しやすい。損失が「観測テキスト上の尤度」なら、それはバグというより目的関数の素直な帰結である。正しさが欲しいなら、現行仕様という別チャネルの検証が要る——それが後の N14 につながる。

よくある混同（in-grain）

  • 「予測だから単純」→ 目的は単純でも、最適化された関数の内部表現は複雑になりうる。単純な目的＝単純な能力ではない。
  • 「予測だから真理不能」→ 不可能の全面宣言ではない。識別条件・実験・外部検証がなければ保証されない、という話である(*9)(*10)。

次の限界

観測尤度だけで「回答機械」を作ると、検索・翻訳・検証・権限が一つの流暢な吹き出しに潰れる。次節で四つの役割を分離する。

3. N3 — 四つの役割：探索・変換・検証・権限

前層の限界 / why_exists

N2 までで「候補を出すエンジン」であることは見えた。だが日常語の「検索」「翻訳」だけで説明すると、Roleが潰れる。Web検索だけが探索ではなく、仕様→コードだけが変換でもない。さらに、候補を現実の結果や行為へ落とすとき、検証と権限が同じ吹き出しの中で勝手に済む、という誤解が残る。有用なシステムは、探索・変換・検証・権限の分離を要する——これが N3 の hard claim である。

仕組み

四役割を、Chat画面の見た目ではなく、システム境界として定義する。

worked example

「このバグを直して」という一文依頼を、四役割に分解する。

  • 探索: 失敗ログ、関連ファイル、既存Testを集める
  • 変換: 仮説→パッチ案、ログ→次の調査手順
  • 検証: 再現Testと関連スイートの結果
  • 権限: ネットワーク制限下の sandbox、workspace外書き込み禁止、本番への到達を承認制にする(*17)

同じ「パッチ提案」でも、検証なし・権限なしならそれはまだテキストである。有用な Agent システムとは、四役割を workflow として設計したものである。

よくある混同（in-grain）

  • 「Toolを付ければ検証済み」→ Toolは探索と行為を増やす。検証は受け入れ条件を別チャネルで測る工程である（N11/N14）。
  • 「人間が最後に見る＝権限」→ 見るだけでは不十分。破壊的操作には明示的な承認境界が要る(*16)。

次の限界

役割を分離しても、機械側が実際に何を最適化しているかが不明なままでは、任せる範囲が決まらない。次節から離散トークンと次トークン損失へ降りる。

4. N4 — Tokenization と next-token 目的：学習空間の定義

前層の限界 / why_exists

N3 は役割を分けたが、モデルが見ている世界の原子がいまだ曖昧である。人間が見る「単語」や「意味」は、学習の直接の入力ではない。テキストはまず有限語彙上の離散トークン列へ写され、その列上で次トークン予測の損失が定義される。Tokenizer + next-token Cross-Entropy が学習空間を定義する——これが N4 の hard claim である。

四役割の話を先に置くと、「では機械は何を探索しているのか」が口語のまま残る。探索の対象が曖昧なままだと、検証も「なんとなく見た感じ」に戻る。だから機構の第一歩は、入力がどの記号列になり、何を正解と呼ぶかを固定することである。

仕組み

Tokenizer は raw text をトークンID列へ変えるアルゴリズムである。トークンは必ずしも単語と一致しない。subword、句読点、空白、コード片が一つのトークンになりうる。SentencePiece は、生コーパスから言語に依存しにくいサブワード分割を学習・適用する枠組みを示した(*2)。同じ内容でも分割が変わればトークン数が変わり、context 消費と計算コストが変わる。

分割規則は「前処理の雑用」ではない。モデルが比較できるのは、最終的に並んだID列の統計である。空白の扱い、数字の割れ方、絵文字、INDENT、日本語の漢字かな混じり——これらは、人間の直感する「一文」とモデルの見る「系列長」をずらしうる。

decoder-only LLM の典型的な事前学習では、位置 t までの文脈から次トークン x_{t+1} を予測する。

  1. Forward で各位置の表現を積み上げ、語彙サイズ分の logit（Softmax 前のスコア）を出す
  2. Softmax が合計1の確率分布へ変える
  3. 正解トークンの負の対数尤度（Cross-Entropy）を損失とする
  4. 誤差逆伝播が損失への寄与を各重みへ伝え、最適化器が少し更新する

学習分布に現れない最新事実や、稀な失敗モードは学びにくい——これは能力不足のレッテル以前に、分布の問題である。

ここでP1が回収され始める: 自信ありげな誤答は、学習空間上で「ありそう」な続きが高い確率を持つことと両立する。尤度の高さと真理は別軸である。

worked example

文字列 oauth2 と、内部で使っている古い商品名が混在する文書を考える。Tokenizer は前者を少数トークンにまとめても、後者を細かく割ることがある。モデルが「覚える」対象は、あなたの意味単位ではなく、分割後の記号列上の統計である。

コードでは、識別子の途中分割が長い文脈を消費し、見た目上短いファイルでも context を圧迫することがある。たとえば VeryLongInternalServiceNameFactory が多数の部分トークンへ割れると、同じ「ファイルを添付した」でも実効の系列長が伸び、のちの Prefill（N9）と Attention の混合（N7）にまで影響する。

別の例として、日付やバージョン番号を考える。人間は 2024-03 と 2026-07 を同じ「日付型」として比較するが、トークン列上では数字とハイフンの並びにすぎない。時系列の新旧が自動で保証されないのも、真理関数ではなく系列尤度だからである。

よくある混同（in-grain）

  • 「トークン＝意味の原子」→ 実装上の離散記号である。意味は後段の表現学習が担う。
  • 「損失が下がった＝正しい知識が増えた」→ 観測分布への当てはまりが良くなった、という主張までが安全である。
  • 「語彙が大きいほど常に賢い」→ 語彙は空間の解像度であり、データと計算の配分（N5）なしに能力は決まらない。

次の限界

離散空間と損失が決まっても、広大なパターンを圧縮する段階が要る。次節は事前学習と計算・データの配分である。

5. N5 — Pre-training と compute–data：広大なパターン圧縮

前層の限界 / why_exists

N4 の目的だけでは「小さなコーパスで次トークンを当てる玩具モデル」も作れる。実務の LLM が必要とするのは、多様な文脈で有用な続きを出せるだけの広範なパターン圧縮である。その段階が Pre-training であり、固定計算予算ではパラメータ数と訓練トークン数の配分が効く——これが N5 の hard claim である(*3)。

仕組み

Pre-training は大量のトークン系列から general-purpose な重みを作る段階である。データ収集、品質フィルタ、重複除去、言語・コード・ドメインの混合物、ライセンスやプライバシー処理が、実際の Training distribution を作る。モデルは世界全体ではなく、この分布から損失を下げるパターンを学ぶ。分布に薄い領域は薄くしか学習されない。

多数GPUでは、Data / Tensor / Pipeline / Expert Parallel などを組み合わせ、勾配の集約（AllReduce 等）で同期する。低精度（BF16、FP8 等）はメモリと演算を節約するが、数値安定性の管理が要る(*18)。並列化は「同じ損失を分散して最小化している」のであり、並列そのものが新しい目的関数を与えるわけではない。

パラメータを増やすほど賢い、という単調神話ではない。固定フロップ予算では、モデルを大きくしすぎてデータを相対的に薄くすると非効率になりうる。Chinchilla 研究は、計算最適の配分を議論するための一次足場である(*3)。以後のアーキテクチャ・データ混合で最適点はずれうる——経験則であり永遠の定数表ではない。

Mid-training（長く続ける事前学習寄り、長文脈・ドメイン混合、継続事前学習など）は製品ごとに置き方が違い、全モデル共通の標準工程ではない。だが思想は同じで、「すでに圧縮した分布を、別の方向へ延長する」である。

P1 の深さ: 事前学習で圧縮された相関が、後から誤りと判明する「ありそうな知識」の供給源になりうる。誤りは後訓練で上書きされることもあるが、保証ではない（→ N6）。事前学習は真実の倉庫ではなく、候補の広大な盆地を掘る工程である。

worked example

同じ計算予算で、(A) 極端に大きなモデルを相対的に少ないトークンで回す、(B) やや小さめのモデルをより多くのトークンで回す、を比較する思考実験をする。Chinchilla の論点は「B側が当時の慣行より良いことがありうる」という配分問題であり、特定製品の正確なトークン数を断言する材料ではない(*3)。

社内適用の例でも同じである。ドメイン特化のために「とにかく巨大モデルを買う」より先に、自社コーパスの品質・重複・期間・ライセンスを見ないなら、薄い分布の上で大きなパラメータを遊ばせることになりうる。

よくある混同（in-grain）

  • 「公開ベンチで一位＝世界知識の網羅」→ ベンチは限定分布上の証拠である。
  • 「パラメータ数が能力の唯一の軸」→ active compute、データ品質、後訓練、推論時計算が別レバーになる（N6, N13）。
  • 「もっとデータを足せばハルシネーションは消える」→ 分布は変えられるが、観測尤度目的そのものは真理保証へ自動変換されない（N2）。

次の限界

広い生成能力ができても、ユーザー意図への追従・選好・検証可能な推論方策までは自動で揃わない。次節が Post-training である。

6. N6 — Post-training：振る舞い分布を動かす別信号

前層の限界 / why_exists

Base model は文章を続けられても、安定して指示に従い、拒否すべきところで拒否し、読みやすい手順を出し、検証可能な課題で長く探索するとは限らない。指示追従・選好・検証可能推論は、別信号で方策を形作る——これが N6 の why_exists である。SFT / 選好最適化 / Reasoning RL は振る舞い分布を動かすが、真理保証ではない——hard claim である(*4)(*5)(*13)。

仕組み

Supervised Fine-Tuning（SFT） は、instruction と望ましい response の demonstration を模倣させる。InstructGPT は human feedback パイプラインの最初に SFT を置いた(*4)。教えるのは主に「この入力形式にはこの出力形式」という行動マッピングである。デモに誤りがあればその誤りも真似る。SFT は文章継続器を指示追従器へ寄せるが、デモ外の事実性や頑健性を保証しない。

選好最適化 は、一つのプロンプトに複数応答を作り、人間または AI judge がどちらを好むかを比較する。古典的な RLHF は選好から Reward Model を学び、PPO 等で報酬を上げつつ元方策から外れすぎないよう制約する(*4)。効きやすいのは、丁寧さ、拒否の仕方、説明の読みやすさ、曖昧依頼への補足である。一方、Reward Model が測っていない事実性や実運用の副作用までは保証しない。

DPO は、preferred の相対尤度を rejected より高める分類型目的で、明示的報酬モデルと online RL を省ける経路を示した(*5)。経路が分岐すること自体が重要である。「RLHFという一語」で後訓練を片付けると、何が報酬で何が上書きされたかが見えなくなる。DPO も選好データの範囲で分布を動かす技術であり、選好が事実確認を含まなければ事実性は直接保証されない。

Reasoning RL / Verifier / Distillation は、数学やコードのように答えを機械採点できる領域で強い。正答や Test pass を報酬にできれば、長い探索トレースを出す方策を強化しうる。DeepSeek-R1 は rule-based reward を含む大規模 RL を報告し(*13)、Qwen3 は thinking / non-thinking と thinking budget を公開した(*14)。これは next-token generator を廃止したのではなく、どの系列を報酬するかと推論時の計算配分を変えている。Verifier が間違えば、物差しに適応する。自由記述や現実因果では報酬設計自体が難しい。

Distillation は、高価な Teacher の応答・推論例・出力分布を使い、軽い Student に似た振る舞いを移す。低コスト化や遅延削減には効くが、Teacher の誤りも引き継ぎうる。

ここでP3が回収される: 「考えた様子」の文章は、後訓練と推論テンプレートで読みやすく最適化されうる出力である。見える CoT は内部思考の完全記録ではない。正しさの証明に使うな。長くなったことが、介入検証（N12/N14）の代替にはならない。

Evaluation・red teaming・release gate は、benchmark スコアがその test distribution 上の証拠であることを思い出す段階である。実世界の全タスク保証ではない。

worked example

spec-to-code 能力は一phaseから生まれない。Pre-training でコードパターンを学び、SFT で指示形式を学び、選好で読みやすさを寄せ、Reasoning RL や Verifier で Test を通す方策を強め、Application 側が repository search と Test 実行を Tool として与える。レビューで「R1 だから正しい」と言うのは、報酬設計と検証チャネルを無視した飛ばし読みである(*13)。

同じモデル名でも、thinking mode の on/off と thinking budget が違えば、比較対象がすでに別レバーになっている(*14)。「先週より賢くなった」談話の前に、後訓練と推論制御のどのノブが動いたかを問うのが N6 の読み方である。

よくある混同（in-grain）

  • 「RLHF＝真理アライメント」→ 選好と安全行動の再配分であり、真理関数の置換ではない(*4)。
  • 「長い thinking ＝深い理解」→ 追加トークンと報酬付き探索であり、因果同定や権限とは別(*14)。
  • 「Post-trainingで世界知識が丸ごと入る」→ 主に方策の再配分である。知識の更新はデータと目的次第であり、保証語にしない。

次の限界

振る舞いが形作られても、一回の応答が文脈のどこを見てどう変換するかが見えない。次節は Transformer ブロックである。

7. N7 — Transformer ブロック：文脈混合と特徴変換

前層の限界 / why_exists

ここまでの節は「何を最適化し、どう振る舞いを曲げるか」を扱った。しかし一トークンを出す瞬間、モデルは文脈のどこを参照し、各位置の表現をどう書き換えているかがまだ空洞である。文脈選択と位置ごとの変換が要る——Attention は文脈混合、MLP は特徴変換であり、literal な外部DB検索ではない(*1)。これが N7 の hard claim である。

仕組み

元の Transformer は、再帰や畳み込みに頼らず、自己注意を中心に系列変換を行うアーキテクチャとして提案された(*1)。現代の多くのチャットLLMは因果マスク付き decoder-only だが、中核の演算は同じ家族である。典型的なブロックでは次が重なる（順序・正規化の詳細は実装差がある）。

  1. Embedding: トークンIDを連続ベクトルへ
  2. 位置情報（例: RoPE）: 相対位置を注意計算で使えるようにする
  3. 正規化: LayerNorm / RMSNorm など
  4. Q/K/V 射影: 隠れ状態を Query・Key・Value へ線形変換
  5. Masked Self-Attention: softmax(QKᵀ/√d)V で、未来を見ずに過去文脈を混合(*1)
  6. 残差接続: 変換結果に入力を足し、情報流と最適化を安定させる
  7. MLP / FFN: 各位置で特徴を拡張・非線形変換・圧縮（SwiGLU 等）
  8. （任意）MoE: Router がトークンごとに少数 Expert を選び、総パラメータと active compute を分離(*12)

直感の対応づけは次である。Query は「今の位置が何を探しているか」、Key は「各過去位置が持つ手がかり」、Value は「選ばれたときに渡す情報」。Attention は類似度から重みを作り Value を混ぜる。MLP は混ぜた情報を位置ごとに変換する。

MHA / GQA / MLA は、主に KV をどう持つかの設計家族である。MHA は head ごとに独立した Key/Value を持つ。GQA は複数 Query head で Key/Value を共有し、推論時の KV を抑える。DeepSeek 系の MLA は Key/Value を latent へ圧縮し、キャッシュ負荷を下げる方向を報告する(*12)。

8. N8 — 自己回帰サンプリング：分布から一つの続きを選ぶ

前層の限界 / why_exists

N7 までで、最終隠れ状態から語彙上のロジットが得られるところまで来た。しかし応答はロジットそのものではなく、選ばれたトークンの連鎖である。温度や Top-p は分布の形を変え、正しさを保証しない——これが N8 の hard claim である。自己回帰依存のため、誤りは後続文脈へ伝播しやすい。

仕組み

LM Head が最終表現を語彙サイズのロジットへ射影する。代表的な選択規則は次である。

  • Greedy: 最大確率トークンを毎回選ぶ
  • Temperature: ロジットを温度で割ってから Softmax。低いほど高確率側へ尖る
  • Top-p（Nucleus）: 累積確率が p に達する最小候補集合からサンプリング
  • Max tokens / stop sequence: 正しさではなく資源と停止条件

学習時は教師トークンへの当てはまりを上げ、推論時は上の規則で一つのサンプル経路を歩む。決定的に寄せても、重みや文脈にある誤りは消えない。

推論の時間構造は二相に分かれる。Prefill は入力系列をまとめて通し、各層の Key/Value を KV Cache へ書く。Decode は新しいトークンだけを通し、過去 KV を再利用する。K トークンの出力には原理的に K 回の逐次ステップが要る。

ここでP2が回収される: Temperature を下げても嘘が消えない。温度は尖りを変えるノブであり、真理ノブではない。低温度は「同じ高確率の誤り」を安定して出しやすくすることすらある。

P1 の再確認: 自信ありげな誤答は、尖った分布のモードとして現れうる。確率の高さと検証済みは別である。

9. N9 — Serving と KV：Prefill/Decode が現実の制約になる

前層の限界 / why_exists

N8 の Decode を「GPUに入れて答えが返る」とまとめると、なぜ長いプロンプトが重いのか、なぜ出力トークンごとにレイテンシが乗るのか、なぜ同時ユーザー数が KV で決まるのかが見えない。実サービスはメモリ帯域と逐次 Decode に縛られる。Prefill で KV を作り Decode で再利用する。最適化の多くは近似や IO 削減である(*6)(*7)(*8)。これが N9 の hard claim である。

仕組み

Application が System Instruction・履歴・取得文書・Tool schema を組み立てる段階で、探索可能な空間が決まる。Orchestrator は検索・再試行・予算・承認・停止を外側で制御する。Agent らしさの多くはこの外側ループから生まれる（詳細は N11）。Model Server では認証・レート制限のあと、Scheduler がリクエストをバッチへ載せる。Continuous Batching は、到着時刻と長さの違う要求を Decode ステップごとに出し入れし、スループットを上げる。

  • Prefill: 入力系列をまとめて通し、各層の Key/Value を KV Cache へ書く。長い入力はここが重い。
  • Decode: 新トークンだけを通し、過去 KV を再利用する。長い出力はステップ数が増える。

KV Cache は「過去を覚えている脳」のロマンではなく、自己回帰のための具体メモリである。要求長が動的だと、素朴な連続確保はない断片化が同時処理数を落とす。PagedAttention は OS のページングに似せ、KV をブロック単位で非連続管理し、断片化を抑える(*7)。

FlashAttention は注意の数学的結果を保ちつつ、タイル化と再計算で HBM–SRAM 間の IO を減らす(*6)。ボトルネックが帯域なら、演算器を増やしても効果は薄い。

Speculative Decoding は小さな draft が複数トークンを提案し、target がまとめて受理／拒否することで、出力分布を保ちつつ逐次性を緩和しうる(*8)。受理率が低いと恩恵は薄い。

量子化 はビット幅を下げて容量と帯域を減らすが、品質・校正・カーネル対応とのトレードオフがある。Tensor Parallel / Pipeline Parallel / Expert Parallel は、巨大行列や層や Expert を複数デバイスへ割り振る。通信が支配すると、見た目のGPU台数は線形に効かない。

ここでP4がさらに回収される: 長い文脈は Prefill 計算と KV 容量・帯域を直接押し上げる。入る上限（context window）と、サービング上「載せる価値があるか」「注目されるか」は別問題である。無関係な長文を貼ることは、見落とし（N7）とコスト（N9）の両方を悪化させうる。

worked example

同じモデルで、(A) 必要な関数だけを添付した短いプロンプト、(B) リポジトリ半分を貼った長いプロンプト、を比べる。B は Prefill が重く、KV が大きく、バッチに載りにくく、かつ Attention 上の distractor が増える。ユーザー体験としては「丁寧に全部入れたのに遅いし的外れ」になりうる。改善は「もっと大きい context の機種へ」だけではなく、探索を絞ることである。

もう一つの例はストリーミング応答である。トークンが順に届く感覚は、Decode の逐次性のUI表現である。途中で止めても「思考の一部」が残るわけではなく、採用済みトークンまでのサンプル経路が残る。ログに残すべきは、流暢な感想ではなく、request ID・レイテンシ・トークン数・Tool call・承認結果である。

よくある混同（in-grain）

  • 「FlashAttention＝近似Attentionで品質低下」→ 代表的な FlashAttention は同一計算の IO 最適実装である(*6)。近似注意は別系統。
  • 「serving最適化＝モデルが賢くなった」→ 多くの場合、同じ方策をより安く早く運ぶ工学である(*7)(*8)。
  • 「context windowの数字＝その長さで常に高精度」→ 上限と品質曲線は別；P4と混同しない。

次の限界

KV と行列演算は、最終的に加速器上の物理制約になる。次節はハードへの写像である。

10. N10 — Hardware mapping：行列積とメモリが支配する

前層の限界 / why_exists

N9 までで「何が重い演算か」は見えたが、なぜ GPU がその形の計算に向くか、学習と Decode でボトルネック感が違うかが抽象のままだと、性能予言と性質がまた混ざる。行列積が加速器上の現実制約になり、GPU は重み／活性の読みと行列積が支配する——N10 の hard claim である(*18)。ここでも、特定チップのランキング予言はしない。再構築するのは資源の種類である。

仕組み

深層学習の支配的カーネルは、しばしば巨大な行列乗累算である。GPU は同種の並列演算を大量にこなし、Tensor Core のような経路が低精度行列積を加速する(*18)。実行性能は演算だけでなくメモリ階層・帯域・カーネル融合に制約される。演算強度（どれだけデータを再利用できるか）が低いと、どれだけ理論FLOPがあっても帯域に頭打ちされる。

学習と推論で資源プロファイルは違う。学習は大きなバッチで並列性が出やすく、勾配同期の通信が壁になりうる。自己回帰 Decode はトークン方向の直列が残り、ステップごとに巨大な重みと成長する KV を読む。だから「同じモデルを学習できるクラスタ」と「低レイテンシで会話できる serving」は、資源の使い方が違う。MoE ではさらに Expert 間の通信が絡む（N13）。

低精度（FP8 等）は、同一アルゴリズムでも実効スループットとエネルギーを変える実装レバーである(*18)。これは「モデルが賢くなった」ではなく、「同じ型の計算をより安い数値表現で運ぶ」話として読む。

ハードウェア層で性質として固定してよいことと、予言してはいけないことを分ける。固定してよいのは、巨大テンソルの移動と行列積が支配的であるという型である。予言してはいけないのは、来四半期のチップ世代がどれだけ安く速くなるか、どのクラウドが最安か、といった外部性能物語である。

worked example

長い出力を待つ対話で、ユーザーは「考えている時間」に見える待ちを体験する。内側では、各デコードステップで巨大な重みテンソルと KV を読み、小さなバッチの行列積を繰り返している、と再記述できる。待ちを減らす手段は「もっとお願いする」ではなく、出力長の上限、draft/検証（投機）(*8)、量子化、バッチ効率、あるいは問題自体を Tool 側へ移すことである。

学習ジョブの例では、GPU利用率が高く見えても通信待ちで実効が低いことがある。一方、対話APIではGPUが「暇そう」でも、個々のユーザーにとっては Decode 直列が支配する。同じハードウェアが、違う仕事では違うボトルネックになる。

よくある混同（in-grain）

  • 「FLOPが多いほど常に速い」→ 帯域や通信が飽和すれば実効は落ちる(*18)。
  • 「ローカルGPU＝全権限で安全」→ ハードウェア所有と権限境界は別（N14）。
  • 「ハードを買えば検証が不要」→ 検証は受け入れ条件の話であり、アクセラレータの話ではない。

次の限界

重みと文脈だけでは「今のリポジトリ状態」「今のテスト結果」は足りない。次節が Tool / Agent ループである。

11. N11 — Tool / Agent ループ：外部探索は権威を自動付与しない

前層の限界 / why_exists

N3 で外部探索の役割を定義したが、機構としては、Tool呼び出しもまた生成トークン（しばしば JSON 等の構造化出力）である。重みにない現在状態は外から足す必要がある。Tool は観測と行為を増やすが、権威を自動付与しない——N11 の hard claim である(*16)(*17)。

仕組み

典型ループは次である。

  1. Application / Orchestrator が Tool schema と権限ポリシーを文脈へ載せる
  2. モデルが Tool call を生成する（変換）
  3. Orchestrator が schema 検証と permission check を行う
  4. Tool adapter が Web / ファイル / shell / DB 等を実行する（外部探索または行為）
  5. 結果を新しい観測として文脈へ戻す
  6. 再度 Prefill/Decode へ入る

モデルが「世界へ直接触れている」わけではない。credential・sandbox・approval・監査ログが実際の authority を持つ(*17)。Agentらしさの多くは、モデル単体の知性というより、この外側の制御ループから生まれる。失敗モードも増える。wrong query、悪い出典、prompt injection、結果の誤読、副作用、長ホライズンの累積誤差である。NIST の生成AIリスクプロファイルが強調するように、リスクはモデル能力だけでなく用途・文脈・人間の役割・ガバナンスに依存する(*16)。

ここでP5が回収され始める: Agent が自信を持って危険な変更を提案する現象は、「悪い心」ではなく、探索・変換が強力なまま、検証と権限が弱いときに起きる。提案の流暢さを承認の代理変数にしてはならない。

worked example

Codex 系のコーディングエージェントが bug を直す流れを、四役割で読む(*17)。

  1. issue を受け取る（目的）
  2. リポジトリ内の関連ファイルを外部探索する
  3. コードと Test を文脈内で関連づける
  4. 失敗仮説をパッチへ変換する
  5. Test を Tool で実行し、介入証拠を得る（ここが検証チャネル）
  6. 失敗ログを戻して再変換する
  7. diff と Test 結果を人間へ提示し、破壊的操作は承認を待つ

「直しました」という自然文だけでは、5と7が欠けていれば証拠は弱い。

よくある混同（in-grain）

  • 「Tool結果は常に真実」→ 取得チャネルの信頼性は別問題。Retrieval と Verification は別工程である。
  • 「エージェント＝自律した責任主体」→ 責任と権限は制度・運用側に残る(*16)。

次の限界

探索と変換が強くなっても、「原因はこれだ」と語れることと、因果を同定できることは別である。次節が因果の階梯である。

12. N12 — 因果の限界：語ることと同定することは別

前層の限界 / why_exists

LLM は「AがBを引き起こす」と流暢に書ける。N2 の観測目的と、N6 の説明文最適化が揃うと、因果語彙は特にそうらしく見える。しかし observational next-token だけでは、因果グラフ・介入・反事実を一意に正しく学ぶことが強制されない(*9)。「原因」と語れることと同定は別——N12 の hard claim である。

N11 で Tool を足しても、この限界は自動では消えない。Test Tool を持っていても、「何を介入として固定し、何を成功とするか」を設計しなければ、モデルは再び因果語彙の流暢な再生産へ戻りうる。

仕組み — Causal ladder

訓練テキストに「Aの後にB」「AとBが一緒に語られる」があっても、交絡・逆因果・選択バイアスを除く実験が含まれるとは限らない。

Transformer が因果表現を絶対に持てない、ではない。observational next-token objective だけでは、因果グラフ・介入・反事実を一意に正しく学ぶことが強制されない。

Causal Parrots は、因果事実を語る成功が、訓練テキスト内の因果言説同士の相関で説明されうる可能性を論じた(*9)。CLadder は形式的因果ルールから大量の問を作り、連想的ヒューリスティックと真正の手順を切り離して評価しようとする(*10)。Joshi et al. は、言及順や時間順を因果と誤認する post hoc fallacy を報告した(*11)。

結論は「LLMは因果不可能」の全面否定ではない。native guarantee がなく、明示構造・実験・Tool・Verifier が必要である。外付けの因果モジュールや実験設計を置けば LLM は仮説生成器として有用になりうる——だがそのとき同定の根拠は外付け側にある。

ここでP3が再回収される: 因果めいた CoT は、Association 階の仮説生成として読む。証明ではない。

ここでP6が深まる: 「学習済みだから因果を理解している」「推論したから原因が確定」は絶対語である。

worked example

Coding Agent が「この race は lock 不足が原因」と言う。それは Association 階の仮説である。該当タイミングを再現する Test を作り、lock 追加だけを Intervention として failure が消えるか確認し、別説明を除くことで因果証拠が強くなる。説明文の自然さは、この実験の代替にならない(*9)(*10)。

ビジネス文脈でも同じである。「施策Xのあと売上が上がった」とモデルがまとめても、それは時間順の連想かもしれない(*11)。対照や段階的導入、交絡の検討なしに施策の因果効果を確定したことにしてはならない。

よくある混同（in-grain）

  • 「Transformerに行動因果がないから因果不能」→ 制約の主座はデータ・目的・識別条件である(*9)。
  • 「ベンチで因果問に正解した＝現場の原因同定ができる」→ スキャフォールドと分布に依存しうる(*10)(*11)。
  • 「因果グラフをプロンプトに書けば同定完了」→ グラフを仮定した推論はできても、グラフ自体の正しさは別証拠が要る。

次の限界

限界が見えたうえで、研究が単一のAGI核心ではなく複数ボトルネックを攻める地図へ進む。

13. N13 — 研究方向：複数制約への攻撃として読む

前層の限界 / why_exists

N6・N9・N12 まで来ると、「次の一位モデル」だけを追う読み方では、何が進歩したのか分からない。進歩は単一AGI核心ではなく、複数制約への攻撃である——N13 の hard claim である。ここは lineage 層であり、性能予言ではなくレバーの地図を渡す。

仕組み — Research map

研究方向を読むときの軸は二つある。(1) モデル内部の表現・計算を触るか、システム外部（Tool・sandbox・評価）を触るか。(2) 学習時の更新か、推論時の追加計算か。同じ「能力が上がった」という語でも、セルが違えば証拠の種類が違う。

DeepSeek-V3 は MoE・MLA・FP8 等で、総パラメータを大きく保ちつつトークンあたりの計算と KV を抑える方向を報告した(*12)。DeepSeek-R1 は検証可能な報酬を軸に長い推論方策を強化した(*13)。Qwen3 は thinking / non-thinking と thinking budget を前面にし、推論時計算をタスクごとに配分する制御を示した(*14)。読むべきは国籍ラベルではなく、どの制約をどの証拠で触ったかである。

worked example

「A社のモデルが考えが深い」「B社は効率が良い」という雑談を、地図のセルへ落とす。thinking budget の有無(*14)、MoE/MLA の有無(*12)、Tool ループの有無、sandbox の有無(*17)、因果ベンチでの評価(*10)——比較軸が違うのに一位表だけを見ると、性質と性能予言が混線する。

もう一つの読み方は失敗談である。reward hacking で長いが空虚な tracing が増えた、sandbox なしで Agent が広い権限を得た、interpretability のデモ図だけ見て配備を急いだ——それぞれ別セルの失敗であり、単一の「AIが暴走した」物語に潰さない。

よくある混同（in-grain）

  • 「interpretabilityが進んだ＝部署で内部を信用してよい」→ 部分観測である(*15)。
  • 「open-weight＝完全再現可能」→ 重みがあってもデータや完全レシピが揃うとは限らない。
  • 「test-time computeを増やせば因果も同定される」→ 追加計算は探索を伸ばしうるが、識別条件そのものは別（N12）。

次の限界

地図を読めても、利用者が今日どう判断するかの規則がまだない。次節が実践の証拠梯子である。

14. N14 — 実践：検証と権限 — 証拠梯子を上がる

前層の限界 / why_exists

N11 と N12 までで、Agent は危険な提案ができ、因果語は仮説にすぎないことが分かった。欠けているのは、利用者が操作する判断規則である。未検証出力 → 出典 → 介入検証 → 承認行為、で証拠強度が上がる——N14 の hard claim である(*16)(*17)。

仕組み

同じ出力文言でも、置かれる段が違う。

  1. 未検証の出力: 「この関数が原因です」は、重みと文脈から作られた仮説である。
  2. 出典つきの仮説: 該当コード・仕様・履歴・ログを示すと探索の根拠は強まる。因果は未確定。
  3. 介入検証した変更: 失敗を再現し、単一変更を入れ、同じ Test が通るなら因果証拠へ近づく。
  4. 承認された行為: diff・Test・残リスクを確認し、人間が権限境界を通したときだけ現実の変更とする。

七つの実務規則：

  1. 探索範囲（repo/URL/file）を先に狭める
  2. 変換契約（入力・出力・非目標・受け入れ条件）を書く
  3. 証拠チャネルを分ける（重み由来／取得出典／Tool実行）
  4. 因果主張は介入（Test・実験）で試す
  5. 権限は自信度で広げない（破壊的操作は別承認）
  6. 説明文ではなく diff・command・log を見る
  7. 成功／予算／停止条件を先に決める

OpenAI の Codex sandbox 開示は、network を default で制限し、workspace 外 write を制限し、unsandboxed action へ approval を要求する設計を説明する(*17)。Sandbox は正しさを保証するのではなく、誤った行為の到達範囲を狭める。NIST のプロファイルは、過信・幻覚・注入などを用途文脈つきで管理する枠を与える(*16)。

目的 → 探索 → 仮説 → 修正 → Test → Diff Review → 承認 → 監視

ここでP5が回収される: 危険な変更提案への対処は、「もっと賢いモデル」以前に、証拠梯子と権限ゲートである。自信度メーターを承認に使うな。

Lab C — Evidence Ladder（教育用トイ）

Evidence Ladder Lab

Hard claim: 同一文言でも、未検証／出典／介入検証／承認の段で証拠強度が異なる。流暢さは段を飛ばさない。

同一主張: 「この lock を削除すべき」
実行してよいか: REJECT — 仮説のみ。流暢さは許可にならない。

Lesson: fluency alone doesn't raise permission（P5）。

静的フォールバック: 段1=reject / 段2=caution（まだ削除不可）/ 段3=caution（本番適用は承認待ち）/ 段4=ok（ゲート通過後）。

worked example

悪い依頼:「login bugを直して。本番も見て」。 良い依頼:「auth で再現する failing Test を先に追加し、public API を変えず最小パッチを作り、関連Testと full suite の結果・diff・残リスクを報告する。production credential と network は使わない」(*17)。

前者は探索範囲と権限が開いたまま変換が走る。後者は契約・検証・権限が先に固定されている。

よくある混同（in-grain）

  • 「人間が最後にYesを押した＝十分」→ 何を見てYesしたかが段を決める。説明だけを見て押すのは段1のまま。
  • 「社内規程があるから機構理解は不要」→ 規程は権限の言語、機構はどこで失敗するかの地図である。両方要る(*16)。

次の限界

地図と規則が揃ったところで、誤解・限界・現象の全回収・自分の立ち位置を固定する。

15. N15 — 誤解・限界・現象回収・立ち位置

Coverage の最終ノードである。ここまでで再構築した鎖を、誤用防止と生活配置へ固定する。

よくある誤解

  • 「次トークン予測だけだから単純である」→ 目的が単純でも内部に抽象特徴や circuit 様の計算が生じうる(*15)。
  • 「探索と変換だけなら新しいものを作れない」→ 新規組合せは生じうるが、因果真理や correctness の保証とは別。
  • 「Chain of Thought は内部思考の完全な記録である」→ 可視化された推論文は出力チャネル（P3）。
  • 「Tool を使えば hallucination はなくなる」→ Retrieval と Verification は別工程（N11）。
  • 「low Temperature なら正しい」→ 低温度は高確率出力を選びやすくするだけ（P2, N8 Lab）。
  • 「long context なら全て覚えている」→ 入ること・重みが乗る・コストが払えるは別（P4, N7 Lab, N9）。
  • 「benchmark が高いから仕事を自律化できる」→ production では permission・可観測性・rollback・説明責任が要る(*16)。
  • 「causal explanation が自然なら cause を理解している」→ Intervention と対立仮説のテストが要る(*9)(*11)。
  • 「学習済み／推論済みだから改ざん不可・絶対確実」→ 観測尤度の性質を万能保証へ読み替える誤り（P6）。

反例・限界

  • このメンタルモデルは LLM 内部の全アルゴリズムを字義どおり記述しない。説明用抽象である。
  • 「因果タスクを一切解けない」とは主張しない。明示構造・実験・Tool・Verifier で構成要素になりうる。同定根拠は外付け側にある(*9)(*10)。
  • 意識や道徳的地位の結論は扱わない。機能的能力と現象的経験は別の問いである(*15)。
  • 閉鎖 frontier の訓練データ・完全アーキテクチャ・内部回路は非公開部分が大きい。公開報告のみを用いる。
  • Lab は教育用トイであり、実モデルの品質・遅延・コストを実測しない。
  • 研究地図は公開情報に基づき更新が必要である。特定時点の一位表を永久化する意図はない。
  • 性能・価格・採用ROIの予言は、再構築する性質の外である。

自分は今、地図のどこに立っているか

読み終えたあなたの仕事は、モデルの順位を暗記することではない。次の問いに、自分の言葉で答えられることである。

  1. 今いじっているのはどの層か — プロンプト（文脈内探索）か、Tool（外部探索）か、サンプリングか、sandbox/承認か、Testか。
  2. 機械に任せているのは探索と変換か — それとも、気づかぬうちに検証と権限まで流暢な吹き出しに委任していないか。
  3. 手元の出力は証拠梯子の何段か — 未検証／出典／介入検証／承認済みのどれか。一段飛ばしをしていないか。
  4. 因果の文を見たら — Association 仮説としてラベルし、Intervention（Test・実験）の設計に変換したか。
  5. 破壊的操作の手前で — 自信度ではなく、権限ポリシーと人間の承認ゲートが仕事をしているか(*16)(*17)。

立ち位置の例:

  • 「自分は今、リポジトリ探索の範囲を設計している（N11の入口）」
  • 「自分は今、Temperatureを触っているが、それはP2の層であり真理層ではない（N8）」
  • 「自分は今、段3のTest結果を見て段4の承認を拒んでいる（N14）」

LLMを理解するとは、答えを信じる方法を得ることではない。どこまでを機械に任せ、どこからを証拠・権限・責任として外側に残すかを見抜くことである。
