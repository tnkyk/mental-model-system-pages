LLMをファーストプリンシプルで理解する

LLMを「答えを知る主体」ではなく、探索、変換、検証、権限の分離として捉え直す。学習から推論、計算資源、因果、研究動向、Codex利用までを一つの流れで理解する。

公開情報の基準日: 2026-07-13 ・ 一次論文と公式資料に基づく

主張する結論

LLMは、重み、文脈、外部情報から役に立つパターンを探し、質問、仕様、コード、画像、Tool結果を別の表現へ変換する仕組みである。真偽、因果、責任は自動では付かない。だから実用では、LLMに探索と変換を任せ、検証、権限、人間の承認を外側に残す。

1. なぜ探索・変換・検証へ分けるのか

LLMが賢く見える理由と、LLMだけでは任せきれない理由は同じ場所から来る。大量の観測文から次にありそうな表現を作る仕組みは、候補を出す力を持つが、その候補が事実か、原因か、実行してよい行為かまでは保証しない。

ここから四つの分離が必要になる。第一に、答えはどこから来たのかを分ける。第二に、表現をどう変換したのかを見る。第三に、候補を証拠で検証する。第四に、現実へ作用する権限を人間とシステム側で制御する。

制約から生まれる四つの役割

観測だけでは世界を保証しない学習データは観測された文章の集まりであり、世界そのものでも実験結果の全体でもない。だから出力はまず候補として扱う。
候補を作るには探索が要る重みに圧縮されたパターン、現在の文脈、Toolで見に行く外部情報から、次の出力に使えそうな材料を集める。
仕事にするには変換が要る質問を回答へ、仕様をコードへ、エラーログを修正案へ、計画をTool呼び出しへ変換することで価値が出る。
現実に出すには検証と権限が要るテスト、出典、制約、レビューで確かめ、削除、公開、支払い、本番変更のような行為は別の承認を通す。

このページで使う用語

重み内探索
重みに圧縮されたパターンから、現在の文脈に合う続きを活性化する。英語では Parametric Search と呼ぶ。
文脈内探索
Attentionがプロンプト、会話履歴、取得文書、Tool結果の関連位置を選ぶ。英語では Context Search と呼ぶ。
外部探索
Web、リポジトリ、データベース、ファイルシステム、実行環境をToolで観測する。
変換
ある表現を別の表現へ写すこと。質問から回答、仕様からコード、コードからテスト、エラーログから修正案、計画からTool呼び出しへの変換を含む。

重要な限定: これはLLMが内部で候補を総当たりしているという文字どおりの主張ではない。Transformer内部には分散表現と大量の行列計算がある。このメンタルモデルは、何を得意とし、何を別途保証しなければならないかを見抜くための道具である。

この結論の根拠

  1. Pre-trainingは主に次のTokenの条件付き確率を最適化し、真実や因果効果そのものを損失関数にしていない。
  2. Attentionは文脈から情報を選び、MLPは表現を変換する。
  3. Toolは外部観測と行為を増やすが、Modelへ自動で責任を与えない。
  4. 因果ベンチマークでは改善例と同時に、時間順序を因果と誤認する失敗も確認されている。

2. 学習: 能力が作られる段階

「LLMを学習する」は単一工程ではない。何を学習可能にするか、広いパターンをどう圧縮するか、どの振る舞いを選びやすくするかは別の段階で決まる。

Pre-trainingの一回の重み更新

  1. Forward Pass: Modelが各位置で語彙全体のLogitを計算する。
  2. Softmax: Logitを合計1の確率分布へ変換する。
  3. Cross-Entropy: 正解Tokenへ割り当てた確率が低いほど大きな損失にする。
  4. Backpropagation: 損失への寄与を各重みへ伝える。
  5. Optimizer: 損失が下がる方向へ重みを少し更新する。

多数GPUではData Parallel、Tensor Parallel、Pipeline Parallel、Expert Parallelを組み合わせる。AllReduceは複数デバイスの勾配を集約する通信操作である。FP8やBF16はメモリと計算を節約する低精度形式だが、数値安定性の管理が必要になる。

3. 推論: リクエストから応答まで

推論は一回のModel呼び出しではない。Application、Orchestrator、Model Server、Runtime、Acceleratorが連なり、Tool呼び出しがあると外部ループへ出て再びModelへ戻る。

4. 因果: 構造上の弱さを正確に言う

LLMが因果らしい文章を生成できることと、データから因果関係を同定できることは別である。

通常のnext-Token Pre-trainingが直接最適化するのは、観測されたToken列の尤度である。学習テキストに「Aの後にB」「AとBが一緒に語られる」という相関があっても、交絡、逆因果、選択バイアスを取り除く実験が含まれるとは限らない。

限定された主張: Transformerが因果表現を絶対に持てないという意味ではない。観測データだけに基づくnext-Token目的関数では、causal graph、Intervention、Counterfactualを一意に正しく学ぶことが強制されない、という意味である。

論文から分かること

  • Causal Parrotsは、因果事実を語る成功が学習テキスト内の因果事実同士の相関で説明できる可能性を論じた。著者のconjectureを含む。
  • CLadderはAssociation、Intervention、Counterfactualを含む形式的な因果問題がLLMに難しい一方、CausalCoTで改善することを示した。
  • Joshi et al. (EMNLP 2024)は、言及順序や時間順序から因果を誤推定するpost hoc fallacyを報告した。
  • Kasetty et al.は、制御されたIntervention benchmarkでGPT系Modelに有望なperformanceも報告した。

結論は「LLMはcausal reasoning不可能」ではない。明示的な因果構造、実験、Tool、Verifierを与えれば改善できるが、native guaranteeはない。

Codexでの読み替え: 「この変更でバグが直る」は仮説である。失敗するTestを再現し、単一の変更をInterventionとして実行し、失敗が消えるか再測定することで因果証拠へ近づく。

6. LLMに対する誤解

「次Token予測だけだから単純」

目的関数が短く書けても、内部に抽象特徴、algorithm-like circuit、planning behaviorが生じうる。仕組みの説明が短いことは、機能が単純であることを意味しない。

「探索と変換だけなら新しいものを作れない」

学習済み表現を制約の下で再構成すれば、新しいprogramやdesignが生じうる。新規性と正しさは別である。

「Chain of Thoughtは内部思考の完全な記録」

見えている推論文は出力チャンネルであり、読みやすく学習されうる。内部activationの完全な記録ではない。

「Toolを使えばhallucinationは消える」

間違ったquery、悪いsource、prompt injection、読み間違い、Toolの副作用が残る。RetrievalとVerificationは別工程である。

「低いTemperatureなら正しい」

同じ高確率出力を選びやすくするだけで、重みや文脈にある誤りは直らない。

「長い文脈なら全て覚えて使える」

Context Windowへ入ることと、必要箇所を正しく利用することは違う。distractor、位置、retrieval品質、KV Cache costが影響する。

「因果説明が自然なら原因を理解している」

因果らしい文章のretrievalかもしれない。Intervention、Counterfactual、alternative hypothesisのTestが必要である。

7. Codex等を使う人の実践ルール

Coding Agentへ任せるのは探索と変換である。正しさを決める証拠、行為を許す権限、結果へ責任を持つ承認は外側へ残す。

  1. 範囲を狭くする: repository、directory、file、non-goalを指定する。
  2. 成功条件を書く: Test名、期待する振る舞い、性能しきい値、browser状態などを観測可能にする。
  3. 正本を指定する: spec、issue、schema、official documentation、既存Testの優先順位を決める。
  4. 探索結果を信頼しすぎない: Web pageやrepository textには古いinstructionやprompt injectionがありうる。
  5. 変換結果をdiffで確認する: 説明ではなくfile diff、command、dependency、artifactを見る。
  6. 因果主張をTestする: 再現、単一変更、再測定で仮説を検証する。
  7. 権限を分離する: network、secret、production、delete、payment、publishは必要時だけapproveする。

同じ出力でも、証拠の強さは変わる

未検証の出力「この関数が原因です」と自然に説明しても、それはまだ重みと文脈から作られた仮説である。
出典つきの仮説該当コード、仕様、履歴、エラーログを示せると、探索の根拠は強くなる。ただし因果はまだ確定しない。
Testで試した変更失敗を再現し、単一変更を入れ、同じTestが通るなら、仮説は因果証拠へ近づく。
承認された行為diff、Test結果、残るリスクを確認し、人間が権限境界を通したときだけ現実の変更として扱う。

依頼例: 「login bugを直して」ではなく、「auth moduleで再現するfailing Testを先に追加し、public APIを変えず最小patchを作り、関連Testとfull suiteの結果、diff、残るリスクを報告する。production credentialとnetworkは使用しない」と依頼する。

OpenAIのCodex sandbox資料はnetworkをdefaultで制限し、workspace外writeを制限し、unsandboxed actionへapprovalを要求する設計を説明している。Sandboxは正しさの保証ではない。誤った行為の届く範囲を狭める境界である。

8. 最終メンタルモデル

LLMは、三種類の探索と多段の変換を行う確率的なエンジンである。重み内探索は重みからパターンを取り出し、文脈内探索はAttentionで現在の入力の関連箇所を選び、外部探索はToolで世界を観測する。TransformerはそれらをTokenやTool呼び出しへ変換し続ける。しかし観測データに基づく学習は、因果、真偽、責任を直接保証しない。Toolを持つLLMを使うときは、Verifierで結果を確かめ、権限で行為を制限し、人間が最終責任を持つ。

有用なAIシステム = 探索 + 変換 + 検証 + 権限 + 人間の責任

LLMを理解するとは、答えを信じる方法を得ることではない。どこまでを機械に任せ、どこからを証拠、権限、責任として外側に残すかを見抜くことである。

このメンタルモデルの限界

  • LLM内部の全algorithmを文字どおりに記述するものではない。
  • LLMがcausal taskを一切解けないとは主張しない。明示的な構造、fine-tuning、実験で改善できる。
  • consciousnessやmoral statusは扱わない。機能的能力とは別の問いである。
  • closed frontier Modelの学習データ、正確なarchitecture、内部回路には非公開部分がある。
