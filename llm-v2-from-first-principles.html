目指す理解

LLM は 条件付き次トークン が支える 検索 と 翻訳 の部品として扱い、できる／できないの線引きと信頼・完了の責任は 窓（Context） と 外側のワークフロー に置く。

この記事がすること／しないこと

すること

  • できること／できないことを、製品カタログではなく第一原理の型から切り出す。
  • 利用者向けに根源機能を 検索＋翻訳 へ圧縮し、その内部実装と限界まで段階的に開く。
  • 窓（Context）の設計と、外側のワークフロー（検証・停止・権限）に信頼と作業を置く使い方まで示す。
  • 層ごとの信頼の置き方と、価値・幸福は人が引き受ける問いであることを境界付きで示す。

しないこと

  • 特定商用モデルの攻略プロンプト、性能予言、投資・GPU購入助言。
  • フレームワーク操作全集、AGIタイムライン、脳科学同一視。
  • 幸福の体系的哲学書化。「検索＋翻訳」を一次論文の用語だと主張すること。

読み方の約束（認知負荷）

  1. 各章は 今の問題 → 必要な部品（最大3つ） → 次の問題 の順だけを進む。
  2. 帯A（日常） と 帯B（機構） を混ぜない。火傷の話の直後に、同じテンションでハード詳細を並べない。
  3. 同じ対象は、下の言い切り辞書の言い方だけを使う。おしゃれな言い換えで迷子にしない。
  4. 入口の道筋は「章の目次」ではなく、これから解く 問題の列 である。

理解の道筋

章の目次ではなく、これから解く問題の列。一度に覚える名前は増やさない。

  1. 上手いのに火傷する——結局、何ができて、どう使えばよいのか。
  2. 「相手」ではなく、何の型として見るべきか。
  3. 利用者にはどう圧縮し、内部では何が起きているのか。
  4. 流暢さ・癖・ぶれはどの層か——遅い・高いはどこに帰属するのか。

CHAPTER 01

上手いのに信用できない日常から問いを立てる

今の問題

火傷は個人技不足だけなのか。

この章の部品（≤3）

  • 裂け目 P1–P3（自信満々な誤答／資料を渡したのに論点落ち／要約は上手いが判断で火傷）
  • 問いの固定：結局何ができて、どう使えばよいか
  • 理解の道筋（この記事が辿る問題の列、≤6）

出口に残す一文（予告）

「結局何ができて、どう使えばよいか」という問いだけが残った。

もう何度も同じことが起きている。チャット欄に質問を投げると、驚くほど滑らかな文章が返ってくる。段落は整っていて、語尾も丁寧で、専門用語まで自然に混ざっている。だから最初の数回は信じてしまう。ところが後になって、その内容が根拠のない思い込みだったと分かる瞬間がくる。数字が違う。存在しない資料名が引かれている。それでいて文章の自信満々さは、正しい答えのときと何も変わらない（*1）。これが最初の裂け目——P1「自信満々な誤答」だ。誤りそのものより厄介なのは、誤りが不安げに出てこないことだ。人間なら分からないときは口調が濁る。ここにはその手がかりがない。

二つ目の裂け目はもっと日常的だ。長い議事録や仕様書をそのまま渡して「要約して」と頼む。返ってくる文章は読みやすく整っている。しかし会議で本当に効いた一言——「この案は予算超過で見送り」——がすっぽり抜けている。渡した資料の量は十分だったのに、肝心の論点だけが素通りされて、代わりに無難な一般論で埋められる。これがP2「資料を渡したのに論点が落ちる」。手元に材料はあった。なのに、材料が「使われた」ことにはならなかった。

三つ目はさらに油断しやすい。「この記事を三行で要約して」「この文章をもっと丁寧な言い方にして」——こういう依頼には驚くほど強い。だから調子に乗って「この二つの案、どっちがいいと思う？決めて」と踏み込む。すると、いかにも判断したような文章が返ってくるのに、根拠を辿るとどちらの案にも当てはまる一般論しか言っていない。決めたように見えて、実は決めていない。これがP3「要約は上手いが判断で火傷する」。上手さと信頼できる範囲は、同じ軸の上にない。

もう一つの日常も同型だ。先週まとめた案件の前提を「覚えている？」と確かめると、前回の結論を自信を持って繰り返してくる。体裁は完璧で、会話が続いているように見える。だが今週入った予算の修正や、口頭で訂正した条件は反映されていない。こちらが「更新した資料はこちら」と貼り直しても、返答の滑らかさは変わらない。この場面はP1–P3のどれとも完全には重ならない——更新された材料と、返ってくる文章の安定さがずれる、という別の裂け目だ。いずれにせよ共通しているのは、上手さの手触りが「ちゃんと見てくれた」という安心を先に作ってしまうことだ。

ここで急いで結論を出したくなる衝動が二つある。一つは「自分の使い方が下手だった」という自己責任オチ。プロンプトをもっと工夫すれば全部解決する、という方向だ。もう一つは逆に「結局これは賢い顔をしただけの何か」という主体への疑いだ。だが両方とも早すぎる。P1–P3はどれも、使い方の技量だけでは説明がつかないし、裏に人格や悪意があるという説明もいらない。必要なのは、そのどちらでもない第三の構え——何をしている装置なのかを、型から見直すことだ。

だから、この記事が最初に固定したい問いは一つしかない。結局何ができて、どう使えばよいのか。 この問い一つを手に持ったまま、次の五つの問いを順番に辿っていく。

  1. 上手いのに火傷する——結局、何ができて、どう使えばよいのか。（今、ここ）
  2. 「相手」ではなく、何の型として見るべきか。
  3. 利用者にはどう圧縮し、内部では何が起きているのか。
  4. 流暢さ・癖・ぶれはどの層か——遅い・高いはどこに帰属するのか。
  5. 何がモデル外か——だから窓の設計と外側のワークフローが本丸なのか。
  6. 層ごとに信頼を置き、何を自分に残すのか。

この六つのうち、機構の名前が出るのはまだひとつもない。P1–P3は後の章で必ず回収する——名前が付き、層が割り当てられる。今はその予約票を切っただけでいい。だが、結局何ができて、どう使えばよいのかという問いだけを手にしても、まだ次の一歩は踏み出せない。答えの型がないからだ。P1–P3のどれを説明するにも、「相手が悪い」「自分が下手」ではなく、何をしている装置かという見方が要る——それがまだ空白にある。民間の「相手」という見立てをいったん外し、次章では「何を扱っているのか」を、もっと単純な見方で一枚に固定する。型が決まって初めて、使い方の議論に戻れる。これが次章への限界である——問いは立ったが、型はまだない。

CHAPTER 02

条件付き次トークンという型へ立て直す

今の問題

民間の「相手」モデルを何で置き換えるか。

この章の部品（≤3）

  • 条件付き次トークン（P(x_t|x_<t) という型）
  • 短いlineage（並列限界→注意の採用）
  • 保証外の先出し（次トークン損失は流暢さしか押し上げない）

出口に残す一文（予告）

扱うのは意味主体ではなく条件付き次トークン分布だと固定できた。

前章で残した問いを、いったん脇に置いてほしい。ここで先に決めるべきは「どう使うか」ではなく「何を扱っているのか」だ。チャット欄の向こうにいるのは、意図を持って会話している「相手」ではない。もっと機械的で、もっと単純な型がそこにある。

その型を一言で言うと、条件付き次トークンである。モデルがやっているのは、これまでに出てきた記号の並び（x_<t）を条件として、次に来る記号（x_t）の確からしさを割り当てることだけだ。数式で書けば P(x_t|x_<t)。この一つの式が、モデルが直接触れているものの全部である（*1）

。文字列の「意味」でも、世界の状態でもなく、扱っているのは有限の記号カタログの上の確率分布だ。会議の資料について聞かれたときも、モデルは「会議の内容を理解して答えを探しに行く」のではない。それまでの文字の並びを条件にして、次に置く文字の確からしさを、ひたすら計算し続けているだけだ。この型を一度固定すると、「賢い相手」という民間フレームが持っていた曖昧さが消える。相手には意図もなければ、隠れた悪意もない。あるのは条件と分布だけだ。

ではなぜこの型なのか。ここで短い必然を一本だけ通す。以前の系列モデル（RNN・LSTM）は、一つ前の状態を計算しないと次の状態が計算できないという逐次依存を持っていた。これは学習時に大きな壁になる——GPUの並列計算を活かせず、長い文章ほど学習が遅くなる。Transformerという設計は、この逐次依存を注意という仕組みに置き換え、再帰も畳み込みも使わずに系列全体を変換できることを示した（*2）。重要なのはここだ。この型が選ばれたのは、モデルが「理解」できるようになったからではない。学習を並列化できるボトルネックの外し方として選ばれたのだ。「なぜこの型で困らなかったのか」という博物館的な年表を辿る必要はない。押さえるべきは「逐次依存という壁があった」「注意でその壁を外した」という一本の対比だけである。

もう一つ、ここで先に触れておきたい保証外がある。この型を学習させるとき、モデルは「次の記号を正しく当てられたか」だけを損失として最小化する。この損失（次トークンの交差エントロピー）は、文章を滑らかに続ける能力を強く押し上げる。しかし、それが事実かどうか、指示にどこまで従うか、誰にどんな権限があるかを、直接保証するものではない（3）。実際、モデルの規模を大きくするだけでは、指示への追従性は本質的には改善しない、ということが確かめられている（3）。この保証外は、後の章で「できないこと地図」として名前と場所を与える。今はまだ名前は要らない。「流暢さを押し上げる目的関数は、それ以外を保証しない」という一文だけを持って先に進めば十分だ。

型が固定できたことで、C1のP1「自信満々な誤答」の輪郭が少し変わって見えるはずだ。誤答が不安げに出てこないのは、モデルが自信を持っているからではなく、次の記号の確からしさを計算するという型そのものに「不安」という状態がそもそも存在しないからだ。

ここで型の輪郭をもう少し丁寧になぞっておく。「条件付き次トークン」という型が言っているのは、モデルの中に何らかの世界モデルや信念の集合があって、そこから答えを取り出しているのではない、ということだ。あるのは、これまでの記号列という条件と、次の記号一つ分の確からしさという出力の対応関係だけである。この対応関係を大量のテキストから学習すると、次の記号を当てる精度が上がるだけでなく、文脈に応じて振る舞いを変える——同じモデルに例を数個見せるだけで新しい形式のタスクにも対応できる——という性質まで一緒に育つことが確かめられている（*1）。ここが「賢い相手」に見えてしまう最大の理由だ。適応しているように見える振る舞いも、型としては同じP(x_t|x_<t)の中で起きている。

型が変わらないまま振る舞いの幅が広がる、という事実は、次の一歩を難しくする。「型は一つなのに、できることは無限に見える」という状態のままでは、利用者は何を基準に「頼める／頼めない」を判断すればいいのか分からない。P(x_t|x_<t) という式は正確だが、業務の場で「これは使える／使えない」を判断する粒度としては細かすぎる。次に要るのは、この型を壊さずに、利用者が意思決定できる大きさへ圧縮することだ。

CHAPTER 03

検索＋翻訳として圧縮する

今の問題

型だけでは使い方が決まらない——利用者粒度はどこにあるのか。

この章の部品（≤3）

  • 検索（材料の関連づけ・取得）
  • 翻訳（表現の変換・生成）
  • 写像先ディスクレーマー（一次用語ではない；検索エンジン／MTと同一視しない）

出口に残す一文（予告）

根源機能を検索＋翻訳と圧縮でき、内部実装への写像先を宣言できた。

P(x_t|x_<t) という型は正確だが、これをそのまま持って「この案件にLLMを使えるか」を判断しようとすると、手が止まる。式は業務の言葉ではない。ここで要るのは、型を壊さずに、利用者が「これは頼める／頼めない」を判断できる大きさまで圧縮することだ。

この記事では、根源機能を二つの言葉に圧縮する。検索と翻訳。この二語で足りる、という主張はこの記事の編集上の圧縮であって、一次論文がそう定義しているわけではない——それは先に断っておく必要がある。

検索とは、窓（今扱っている材料の置き場）や外部の索引の中から、材料を関連づけて拾うことを指す。この記事での写像先は二つある。内部では、モデルの中の注意という仕組みが、いま生成しようとしている位置と、これまでの材料の各部分との関連度をソフトに計算し、関連の強い部分を重く拾い上げている（1）。数式や層の積み方はまだ開かない——ここでは「内部にも検索に相当する働きがある」という宣言だけで十分だ。外側では、モデルの外にある文書索引から関連文書を取り出し、それを生成の条件として渡す、という構成が別に存在する（2）。これも同じ「材料を関連づけて拾う」という機能に写せる。内側と外側、どちらも検索という同じ言葉の下に置ける。

翻訳とは、拾った材料と条件から、別のトークン列へ表現を写すことを指す。要約、言い換え、丁寧語への変換、続きの生成——これらはすべて、条件付き次トークン分布から新しい記号列を生み出すという同じ土台の上にある（*3）。「契約書の第3条だけ抜き出して現代語に直して」という依頼は、この二つにきれいに分解できる。まず第3条という材料を関連づけて拾うのが検索、それを現代語という別の表現へ写すのが翻訳だ。二つの機能が別の箱に入っているとわかると、依頼のどの部分をLLMに投げているのかが見えるようになる。

同じ分解を、業務の依頼にも当てはめられる。

依頼文の断片検索か翻訳か
「社内規程の第7章から該当条項を拾って」検索（材料の関連づけ）
「取引先向けの丁寧語に直して」翻訳（表現の写し）
「A案とB案のどちらを採用すべきか結論を出して」どちらでもない（採否は型の外）

表が手元にあると、依頼のどこまでがLLMの箱に入り、どこからが人の判断かを、会議の場で指差しできる。

ここで一つ、はっきり境界を引いておきたいものがある。C1で挙げたP3——「調べて要約」は上手いのに「判断して決めて」で火傷する——は、この検索と翻訳のどちらにも入らない。決めるという行為は、材料を拾うことでも、表現を写すことでもない。翻訳が上手ければ判断も上手くいく、というのは誤った期待だ。翻訳の上手さは表現の写し方の上手さであり、採否を決める責任を負う上手さではない。ここを混ぜないことが、この先ずっと効いてくる。

最後にもう一度、圧縮の限界を確認しておく。検索と翻訳という言葉を、そのまま検索エンジンや機械翻訳という既存の製品カテゴリだと思わないでほしい。名前が似ているだけで、内部で動いている仕組み（注意によるソフトな関連づけ、条件付き生成）は別物だ。この二語は、利用者が判断するための足場であって、内部実装の正式名称ではない。さらに言えば、この圧縮は「LLM＝検索エンジン＋翻訳機」という製品説明にもならない。棚のカテゴリで買い物する話ではなく、内部の部品を見える化するための編集上の二箱だ。もっと言えば、一次の論文群のどこにも「LLMは検索と翻訳の二つでできている」とは書かれていない。この記事が使い方の判断を早めるために選んだ圧縮であり、内部の積層はもっと細かく、もっと入り組んでいる。細部を知りたくなったら、いつでもこの二語から元の実装へ戻れるようにしておく——それがこの圧縮の役割だ。

二つの箱の輪郭がつかめると、C1のP3の境界がもう一段はっきりする。「調べて要約して」という依頼は検索＋翻訳の組み合わせにきれいに収まるからよく効く。「決めて」という依頼は、検索でも翻訳でもない第三の行為——採否の責任を引き受けること——を求めている。その行為をどこに置くかは、この記事のもっと後の章で名前を与える。ここではまだ名前は出さないが、「検索でも翻訳でもない場所がもう一つ要る」という予感だけ持って進んでほしい。

これで、検索と翻訳という二つの箱は手に入った。しかし箱の中身はまだ空だ——注意がどう重みを付けるか、表現がどう1語ずつ伸びるかといった話は、まだこの二語だけでは説明できない。次に要るのは、その箱の中で実際に何が起きているのかを、「切る」「関連づける」「順に出す」という三部品まで手で触れるところまで開くことだ。

二つの往復経路と律速の違いを地図として持てれば、この章は十分。

学習時の往復：目的から並列ハードウェアへ

学習中のAIが「大変」なのは、次トークンを当てる目的関数（交差エントロピー）を大量のテキストに対して最小化する計算が、そのまま巨大な行列演算（GEMM）の連続に落ちるからだ。GEMMは入力・出力のサイズと演算数の比率（算術強度）によって、演算そのもの（math-bound）で待つか、メモリ転送（memory-bound）で待つかが切り替わる(*1)。学習の大バッチ行列積は算術強度が高く、GPUを計算で埋めやすい——ここまでは「賢さ」の話ではなく、線形代数の資源配分の話である。

しかし十億〜千億パラメータの重みとその勾配・オプティマイザ状態は、単一デバイスのメモリに収まらない。そこでモデルをレイヤ内で分割し、複数デバイスで一つの巨大な計算を協調させるモデル並列が要る(*2)。データ並列を素朴に重ねるだけでは各デバイスがオプティマイザ状態を丸ごと複製してしまい、メモリの冗長が学習可能なモデル規模の壁になる。ZeROはこの冗長を分割し、デバイス数に比例して学習可能な規模を伸ばす(*3)。さらに、注意計算そのものの壁時計時間は演算数だけでなくGPUの高帯域メモリ（HBM）とオンチップメモリ（SRAM）の間のデータ移動（IO）に律速されやすいことが分かっており、FlashAttentionはタイリングでこのIOを削って学習と長文脈処理を加速する(*4)。

つまり学習時の往復は、「目的関数（次トークンを当てる）→ 大粒度の行列演算 → 単一デバイスを超えるメモリの壁 → モデル並列とZeROによる分割 → 注意計算のHBM↔SRAM IOの削減」という一本の経路である。ここでボトルネックになるのは「モデルの理解力」ではなく、デバイス間通信・メモリ容量・メモリ帯域という物理的な資源だ。

推論時の往復：条件からKVと帯域へ

チャットでの「重さ」は別の場所にある。推論では学習のような教師強制の並列処理ができない——1トークンを出すごとに、それまでの条件付き次トークンの分布をもう一度計算し直す必要があるからだ。この再計算を避けるため各トークンのkey/valueを保持して再利用するのがKVキャッシュだが、このキャッシュはリクエストごとに巨大で、会話が伸びるたびに動的に増減し、断片化がバッチサイズ——同時に何人分の会話を捌けるか——を制限する(*5)。増分デコードのたびにこの大きなkey/valueテンソルを再ロードすること自体がメモリ帯域の律速になりやすいことも分かっており、Multi-Query Attentionはヘッド間でkey/valueを共有してこの帯域要件を減らす(*6)。速度と品質を両立させたい場合の中間案として、Grouped-Query Attentionがある(*7)。

行列演算の算術強度という観点で言えば、デコードの各ステップは実質「行列×ベクトル」に近い形になりやすく、これは常にメモリ律速になる(*1)。つまり推論のデコード段は、学習の大バッチ行列積とはまったく別の場所で待たされている。この逐次性そのものを迂回する工夫として、小さな近似モデルの提案を大モデルが並列に検証し、出力分布を変えずに複数トークンを一気に進める投機的デコードがある(*8)。また、学習済みの重みを再学習なしに低ビットへ圧縮するポストトレーニング量子化（GPTQ）は、メモリ占有量を減らし単一デバイスでの推論を可能にする(*9)——これは学習後・提供前の層で起きる、また別の資源配分の工程である。

つまり推論時の往復は、「条件付き次トークンからの切り出し → KVキャッシュという会計 → バッチ断片化と帯域律速のデコード → 投機的デコードや量子化による緩和 → 提供システムのページング」という経路であり、学習時の往復とはボトルネックの位置も解決の道具もまったく別である。

真正性・権限・完了・価値選定は外側だと分かれば、この章は十分。

できないこと地図

条件付き次トークンの型は、次にどの記号が来やすいかを割り当てるだけで、その記号列が事実かどうか、誰の許可を得ているか、仕事が完了したかを判定する仕組みを持たない。目的関数（次トークンの交差エントロピー）は流暢な継ぎ接ぎを押し上げるが、事実性・意図・権限を直接最適化しない。ここから導かれるできないこと地図は四項目——真正性（言っていることが本当か）、権限（その操作をしてよいか）、完了（仕事がもう終わったか）、価値選定（どの目的を追うべきか）。これらはどれもモデル内部の次トークン損失では定義されていない。

規模を大きくすること自体もこの地図を埋めない。人間フィードバックを使った適応（SFT→選好ランキング→強化学習）の landmark 研究は、パラメータ数の小さいモデルが人間評価でパラメータ数の大きい素の事前学習モデルより好まれる場合があることを報告しており、これは「意図に従う」という性質が「大きい」という性質とは別の軸にあることを示している(*10)。同じ研究は、適応後も単純な失敗が残ることも記録している——適応は完全な置換ではなく、できないこと地図を消す工程ではないという点も、この章で確認しておく価値がある。

代理指標は本人ではない

ガードレールも同様に二層に分かれる。訓練時に原則リストへの自己批判と改訂・AIフィードバックを通じて規範を内在化させる層(*12)と、推論時に入出力を分類し違反カテゴリを列挙して遮断する専用の外側ゲート(*13)。どちらも「規範を守るモデルらしい振る舞い」を強めるが、規範の遵守そのものを保証する第三者検証ではない。そして規模を大きくするだけでは意図追従は本質的に改善しないことが実証されており(*10)、大きい＝安全でも大きい＝従順でもない。

主体誤認をもう一度断つ

意見箱（この記事の判断規則には採用しない）：「次トークンを十分に予測することは、そのトークンを生んだ背後の現実を理解することだ」という強い解釈がある(*14)。単なる模倣を超えて洞察ある振る舞いへ外挿しうる、という主張も同じ立場から来る。

この記事はこの解釈を判断規則には採用しない。かわりに、形式的に妥当な文字列を継ぎ接ぐ能力と、コミュニケーションの主体であることを混同しないという規範的な警告のほうを採る(*15)。この警告は査読を経た規範的枠組みとして提示されたものであり、大規模言語モデルのベンチマーク上の進度を、そのままコミュニケーションの主体性や理解の証拠とみなすことの危険を明示している。流暢さ・ベンチ順位・ガードの表示のいずれも、「主体が理解している」という結論を保証しない。できないことは製品の欠陥ではなく、型そのものの外側にある。

この二つの意見を並べて置くのは、優劣を裁定するためではない。次トークン予測が統計的に何を圧縮しているか（事実）と、その圧縮から何を主張してよいか（意見の割れる場所）を、はっきり分けて持ち歩けるようにするためだ。この記事の判断規則は後者についてBender側を採るが、次トークン予測そのものの威力を否定するわけではない。

だからこそ、次に効くレバーは「もっと賢いモデルを選ぶ」ではなく、モデルの外側——何を窓に入れ、どう確かめ、どこで止めるか——にあるはずだ。

CHAPTER 08

窓（Context）の設計

今の問題

資料を渡したのに、大事な論点が抜けた要約が返ってくる。プロンプトの書き方をどれだけ工夫しても、これだけでは解決しない場面がある。

この章の部品（≤3）

  1. 有限窓——一度に扱えるトークンの量には限りがある
  2. 位置効果——窓のどこに置くかが使われ方を左右する
  3. high-signal集合——「たくさん貼る」ではなく「効く分だけ置く」

出口に残す一文（予告）

窓（Context）は検索材料の効用設計であり、入れた＝使えるではない、と言えれば十分。

窓（Context）とは何か

モデルが一度のサンプリングで参照できるトークンの集合を、この記事では窓（Context）と呼ぶ。指示・会話履歴・貼り付けた資料・検索結果・ツールの出力——これらはすべて同じ窓に、同じ型のトークンとして並ぶ。窓は条件付き次トークンの「条件」側そのものであり、条件に何を積むかによって出力が変わる。窓は有限であり、内部の注意という関連づけ（検索）は、窓の中のどこに何を置くかによって実際に使われる度合いが変わる。プロンプトエンジニアリングが単発の呼び出しにおける指示・例・出力形式の書き方を最適化する技法だとすれば(*16)、窓の設計はその外側——履歴・外部データ・ツール出力まで含めた、窓に入れるトークン集合そのものの管理である(*17)。

窓が有限であることには理由がある。内部の注意という関連づけは、窓の中のトークン同士をすべて結びつけて重みを計算するため、窓を長くするほど関連づけの計算量は急に増える。だから「窓を無限に長くすれば全部解決する」という発想自体が、注意という仕組みの成り立ちと矛盾する。窓は増やせば増やすほど良いリソースではなく、増やすほど中身の管理が難しくなる有限リソースとして扱う必要がある。

たとえば、100ページの仕様書をそのまま窓に貼って「この矛盾点を洗い出して」と頼む場面を考える。窓には仕様書の全トークンと指示が同じ型で並ぶが、注意という関連づけは仕様書の中の遠く離れた2つの節を等しく強く結びつけてくれるわけではない。結果として、実際に矛盾を突いてくれるかどうかは、その2つの節が窓のどこに位置しているかに強く依存する。

入れた＝使えるではない

high-signal集合という設計

したがって窓の設計は「量を増やす」から「効く分だけを、効く場所に置く」への転換になる。実務的な手がかりは三つに絞れる。

第一に、資料全体を貼るのではなく、その仕事に効く最小の集合（high-signal集合）を選ぶこと。100ページの仕様書ではなく、矛盾が疑われる章だけを検索で取り出してから窓に入れれば、注意という関連づけが働く範囲そのものが絞られる。

第二に、特に重要な指示や結論は窓の先頭または末尾に置くこと——真ん中は最も見落とされやすい位置だからだ。長い会話履歴の途中に重要な決定事項を埋め込むより、要点だけを先頭に再掲するほうが、位置効果の前では有利になる。

第三に、長い会話や作業ログでは、古い情報を要約・退避させて窓を定期的に整理すること。エージェント的な複数ターンのやり取りでは、窓に入る候補（履歴・ツール出力・検索結果）がターンを追うごとに膨張しやすいため、周期的なキュレーションを組み込まないと、有効に使える窓の割合がターンを重ねるごとに下がっていく(*17)。

これらはいずれも、モデルの「賢さ」を上げる操作ではなく、有限の注意予算をどこに使わせるかという設計の操作である。

削る勇気

窓の設計で最も効くのは、多くの場合「足す」ではなく「削る」判断だ。長い議事録をそのまま貼るのではなく、論点だけを先頭に抜き出してから貼る。関連しそうな資料を全部つなぐのではなく、今の問いに直接効く節だけを検索で取り出してから窓に入れる。「たくさん貼れば取りこぼしがない」という直感は、位置効果の前では裏切られやすい。窓を厚くする前に、削る・外側の検索に任せる・先頭や末尾に動かすという三つの選択肢を先に試す価値がある。

窓の設計だけでは業務は閉じない

ここまでで、一回のやり取りをどう組み立てるかは見えた。しかし窓を最適化しても、「この回答で本当に決めていいか」「この作業はもう終わったか」を判定する仕組みはまだ窓の外にある。窓の設計がうまくなるほど、一回の答えの質は上がるが、それを何回積み重ねて一つの仕事を終わらせるかは、まったく別の設計問題として残る。単発の窓最適化の先に、複数回のやり取りをどうつなぐかという設計が残っている。

CHAPTER 09

外側のワークフロー

今の問題

検索やツールをつないでエージェント化すれば、あとの検証や判断まで任せてしまえるのだろうか。一発のプロンプトに賭ける代わりに、何を外側に用意すればよいのか。

この章の部品（≤3）

  1. 検索／翻訳段への配置——AIはワークフローのどの箱に置かれる部品か
  2. reason–act–observeループ——推論と行動を交互に回す基本形
  3. 外側の検証・停止・権限——完了と信頼はモデルの外で決める

出口に残す一文（予告）

AIは部品であり、完了・検証・停止は外側のワークフローだと言えれば十分。

AIは部品として配置される

業務を単純化したうえで、AIを置けるのは検索（材料の取得・関連づけ）と翻訳（表現変換）の箱だけだと線を引く。RAGはこの配置の原型で、パラメータ内の知識だけでは知識集約タスクの更新や出典提示に限界があるため、外部の非パラメトリック記憶を検索し、その結果を生成の条件として結合する(*19)。ツール呼び出しも同じ構造で、モデルはいつどのAPIを呼び、どんな引数を渡し、結果をどう次の生成に取り込むかを自己教師で学べるが、実際の実行と結果の返却は外側のシステムが担う(*20)。現行の製品APIでは、モデルがツール呼び出しを返し、アプリがそれを実行して結果を返し、モデルが最終的な文を出すという往復として定義されている(*21)。

ここで重要なのは、ツールの実行結果も窓に戻ってくる時点では他のトークンと型上区別されない、ということだ——検索結果も計算結果もツールログも、モデルにとっては条件付き次トークンの条件でしかない。たとえば「在庫システムに問い合わせて、残数を返す」というツールを繋いだ場合、モデルが受け取るのは「残数は12」という文字列トークンであり、それが本当に今の在庫を反映しているか、問い合わせ自体が正しい商品IDに対して行われたかは、モデルの内部には保証する手段がない。ツールを一つ繋ぐたびに、その出力を検証する責任もひとつ増える、という捉え方のほうが実務に近い。

reason–act–observeという基本ループ

一回の自己回帰呼び出しは世界を更新しない。窓に積まれた条件から次トークンを出すだけの手続きは、その出力が環境に何をもたらしたかを自分では観測できない。推論トレースと行動を交互に生成し、行動で得た観測を次の推論に戻すループ（reason–act–observe）を組むことで、外部接地のない静的な思考連鎖より、幻覚が減り状況に適応しやすくなることが示されている(*22)。このループ自体は「考えて、動いて、確かめて、また考える」という単純な反復だが、単純であることが利点でもある——各ステップで何が起きたかを人間が追いやすく、どこで止めるかを差し込みやすい。

Lab — ツールと権限の境界教育用トイ

確かめること： ツール呼び出しや検索結果の挿入は文脈への追加トークンでしかなく、モデルは呼び出し権限や検索結果の真正性を機構的に検証しない；検証と権限管理は外側のシステムが担う。

ツールの結果が窓に戻ったあと、それが「検証済みの事実」ではなく「もう一つの条件トークン」として扱われる境界を実際に動かして確かめると、接続することと信頼できることの違いが手で分かる。この基本ループの上に、決定論的なコードが手順を制御するworkflowと、LLMが動的に手順を決めるagentという区別があり、複雑さは測定可能な改善がある場合にだけ足すべきだ、という設計原則がある(*23)。長時間動くエージェントでは、窓（Context）がセッションをまたぐと記憶が消えるため、進捗ファイルやチケットのような「窓の外の状態」でセッション間を橋渡しし、検証可能な完了条件を明示する必要がある(*24)。有限コンテキストをOSの階層メモリに見立て、必要なときだけ外側の記憶とやり取りするページング的な発想も、同じ「窓の外に状態を置く」設計の系譜にある(*25)。

配線と検証は別のプロトコルの仕事

ツール・検索・外部リソースをどう窓に流し込むかを標準化する配線契約として、Model Context Protocol（MCP）のようなプロトコルがある。MCPはHost/Client/Serverという参加者と、tools・resources・promptsといったデータ層のプリミティブを定義し、どのツールがどんな形で窓に材料を渡せるかを標準化する。しかしMCPの公式アーキテクチャ自身が明示するように、これはコンテキスト交換の配線に限定されており、アプリがLLMをどう使うか、受け取ったコンテキストをどう管理し真正性を検証するかは規定しない(*26)。つまり、ツールやMCPで「つながっている」ことと、その出力が「正しい」「権限内である」ことは、まったく別の層の話だ。検証・停止・権限は、配線プロトコルの外側で、アプリケーションと人間が引き受ける手続きとして設計するしかない。

職能が外側のワークフローに移る

一発のプロンプトに賭けるのではなく、業務を単純化し、AIを検索／翻訳の段に配置し、完了の定義・検証の手順・停止条件・誰が最終承認を出すかを外側のワークフローとして先に決めておく——この順番に職能が移る。エージェントを繋ぐこと自体は魔法ではなく、繋いだ先の検証と停止をどう外側に置くかの設計問題である。

検索・翻訳・窓・外側のワークフローという地図は、これでひととおり揃った。ここまでの層はすべて「AIをどう配置し、どう検証し、どこで止めるか」という設計問題であり、その設計を担うのは依然として人間である。あとは、この地図のどこに自分が立ち、何を自分の判断として引き受けるかだけが残る。

CHAPTER 10

層で置き、開かれた問いを残す

今の問題

地図とレバーは揃った。では、どこに立ち、何を自分の判断として引き受けるのか。

この章の部品（≤3）

  1. 層別信頼チェック——どの層に何を委ねてよいか
  2. 現象の総回収——冒頭の火傷体験を層で説明し直す
  3. 持ち帰り一文と開かれた問い——モデル一文と、価値・幸福という残る問い

出口に残す一文（予告）

専門家級の判断地図を持ち帰り、価値・幸福は自分に残る問いだと閉じられれば、この記事は十分。

最終モデル：一文

LLMは条件付き次トークンが支える検索と翻訳の部品として扱い、できる／できないの線引きと信頼・完了の責任は窓（Context）と外側のワークフローに置く。これがこの記事全体の持ち帰りである。判断規則を四つに圧縮すれば——(1) 依頼を見たら型で切る：これは検索か翻訳か、それとも型の外（真正性・権限・完了・価値判断）か。(2) 翻訳に真偽裁定・採否決定・権限行使・完了宣言を載せない。(3) 失敗したらモデルの人格ではなく層を当てる——窓・目的関数／適応・切り出し・学習または推論スタック・外側のワークフローのどれか。(4) 窓を増やす前に、削る・外側の検索に移す・先頭や末尾に置くを試す。この四条項の背後には、能力を一次元の「生の賢さ」だけで測らない、相補的な律速要因を見るという判断パターンもある(*27)。

実務で立ち位置を決めるとき、次の五つを順に確認するだけでよい。(1) 型——今の依頼は検索か翻訳か、それとも型の外か。(2) 目的関数と適応——流暢さや口調の癖はどこまで委ね、どこから自分で直すか。(3) 切り出し——同じ質問でぶれるのは性格ではなく手順の層か。(4) 窓（Context）——渡す材料は high-signal か、論点は先頭か末尾に置いたか。(5) 外側のワークフロー——検証・停止・完了宣言は誰が引き受けるか。この五項は、失敗したときに「AIがわるい」のではなく層を当てるための最低限のチェックだ。

冒頭で並べた火傷の体験は、すべて層に帰属できる。

現象帰属する層
P1 自信満々な誤答流暢さは目的関数の副作用であり主体的な理解の証拠ではない
P2 資料を渡したのに論点を落とす注意という関連づけの位置依存性と窓（Context）の設計
P3 「調べて要約」はできるが「判断して決めて」で火傷翻訳に採否決定を載せない境界（できないこと地図）
P4 長いと遅く、同じ質問でぶれる切り出し手順と学習・推論それぞれのスタック往復
P5 ツールをつなげても検証が自分に残る外側のワークフローの検証・停止
P6 一発プロンプト依存で業務設計が後回しになる外側のワークフローの設計が先にあるべきこと
P7 出力量だけ増えて「良い仕事」が見えにくい技術地図の外にある、人が引き受ける問い

どの火傷も「AIの限界」という一言ではなく、具体的な層の名前を持てる。流暢さの副作用と、注意の位置依存性と、外側の検証の欠落は、まったく別の対処を要求する別の問題であり、その区別自体がこの記事の持ち帰りである。同じ症状に見えても、原因の層が違えば打つ手も違う——流暢な誤答には切り出しや検証の追加が効き、論点落ちには窓の設計が効き、検証の欠落には外側のワークフローの再設計が効く。一つの対処法を全ての火傷に当てはめようとする発想そのものが、層を分けずに「AIの中で何かが起きている」と一括りにする誤りに戻ってしまう。

層で立ち、外側に何を残すか

条件付き次トークンという型、検索と翻訳という圧縮、窓の設計、外側のワークフロー——これらは実務の判断地図として持ち歩ける。読者は今、自分が扱っている案件がこの地図のどの層の問題かを、比較的具体的に指し示せる位置に立っている。「賢さ」を一次元で測ろうとする代わりに、外界の速度・データの質・制度の整備といった相補的な要因のどこが今のボトルネックかを見る、という判断パターンも、この地図の使い方の一つだ(*27)——ただしこの判断パターンは意見であり、それが導く「強力なAIの先にある未来像」まで一緒に採用するわけではない。

だが、この地図はどの目的を追うべきか、どれだけ自動化してよいか、生産性という言葉だけでは測れない「良い仕事」「良い生」が何かまでは答えない。流暢な文章を大量に生み出せるようになったからといって、それが良い仕事や良い生を自動的にもたらすわけではないし、モデルの出力を主体の理解だと錯覚しないという規範(*28)は、自分自身の価値判断まで機械に委ねてよいという話には拡張できない。出力の量だけが増えていくなかで、「どの仕事を残し、何を手放すか」「誰のために時間を使うか」——価値と幸福は、技術地図の外にある人間の問いとして開いたままにする。それは今すぐ答える必要のある問いではないし、この記事が体系的な答えを持っているわけでもない——ただ、この地図を手にしたあとに、その問いだけがはっきり残る、というところで筆を置く。

持ち帰るメンタルモデル

一文の芯

LLM は 条件付き次トークン が支える 検索 と 翻訳 の部品として扱い、できる／できないの線引きと信頼・完了の責任は 窓（Context） と 外側のワークフロー に置く。

判断規則

  1. 上手く見えても、まず検索材料は十分か／翻訳依頼は能力内かを点検する。判断・決裁・真偽裁定を翻訳に載せない。
  2. 失敗したら人格ではなく層を疑う（窓／目的・適応／切り出し／KV・提供／外側検証）。
  3. 信頼は層ごとに置く。流暢さ≠根拠、ツール≠検証済み、ガード≠完了。
  4. 利用設計の既定順は、単純化 → 検索／翻訳段にAI配置 → 完了・検証・停止・権限を外側に残す。
  5. 窓（Context）を増やす前に削る／移す／先頭・末尾に置く。「入れた＝使える」を仮定しない。
  6. 遅い・高い・長い壊れは、学習と推論を混同せず、推論ならKV／decode帯域／提供を先に疑う。
  7. ベンチやLLM-as-judgeで賢さを一次元化しない。
  8. 「予測＝理解」を業務判断に使わない（意見として知るだけ）。
  9. 価値・幸福・何を自動化しないかの選定はモデル外の人間の問い。

モデルが外れるとき

  • 検索＋翻訳は教育用圧縮であり、検索エンジン／機械翻訳製品との同一視で外れる。
  • 短い窓・ツールなしでも実用になる場面はある——それでも保証の層構造は消えない。
  • 組織の監査制度・法的責任は本モデルの外に残る。
