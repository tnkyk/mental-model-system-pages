Bitcoin from First Principles

価格物語ではなく、二重支払いを止めるために署名・UTXO・Proof-of-Work・フルノード検証がなぜ必要かを、現象から再構築する長文ガイド（v2 deepened）。

Mental Model Studio · 2026-07-14 · v2 · 投資助言ではない

最終メンタルモデル

Bitcoinは、署名で「誰が許可したか」を示し、UTXO台帳で「何がまだ未使用か」を示し、Proof-of-Workの連鎖で「どの履歴が正規か」を示し、フルノードが同じ規則で検証する、信頼最小化の電子現金システムである。価格や取引所残高は、その上に乗る市場・委任レイヤである。

あなたがすでに触れている三つの不安

ウォレットの復元画面で、12語のシードフレーズを入力しようとして、古いメモの順序が合わない。アプリは淡々と「無効なチェックサム」とだけ言う。サポート窓口はない。銀行なら「本人確認の書類を」と言われるが、Bitcoinプロトコルには「本人」という概念がない。鍵を失えば、その鍵に紐づくUTXOは誰のものにもならない。永久に動かせない。これがP4の正体である——あなたが主権者であり、救済当局でもある。

別の場面。取引所アプリに「残高 0.42 BTC」と表示されている。送金ボタンを押すと「処理中」になる。同時刻、友人は「まだ届いていない」と言う。取引所の画面は内部帳簿上の負債（IOU）を見せているだけで、あなたが秘密鍵を持っているわけではない。取引所が破綻したら、その数字は債権争いになる。自分の鍵で署名できるUTXOと、取引所の残高表示は、同じ「BTC」というラベルでも別レイヤである。

三つ目。高額の支払いを受け取った相手が「6確認待ってから商品を渡す」と言う。あなたは「反映済み」と表示された時点で安心したが、相手は別の時計を見ている。数分後、ネットワーク上で別の枝が伸び、あなたの送金が一時的に「なかった枝」に載っていたら——商品だけ先に渡してしまう。確定は感覚ではなく、累積Proof-of-Work（PoW）の上に積まれたコストの話である（P1, P3）。

これらは投資の成否ではない。あなたがすでにBitcoinの周辺で感じている主権・委任・確定の三つの裂け目である。

この記事がすること／しないこと

すること：上の三つの不安を再構築可能なメカニズムへ分解する。なぜ銀行のような台帳係なしに「同じコインを二度使えない」と主張できるかを、制約→必然の鎖で説明する。各層でインタラクティブ Lab（UTXO・PoW・Validation・Fork choice）と図へ接続する。

しないこと：価格予測・売買タイミング・ポートフォリオ助言。Lightning Network やサイドチェーンの詳細。Script / Taproot のオペコード一覧。

読み終えたとき、あなたは「BTCが上がるか下がるか」ではなく、「自分が今、鍵・台帳・確認のどこに立っているか」を説明できるはずである。

最初の問い

なぜ、画面上の数字の移動が「現金を渡した」ことになりうるのか。そして、なぜ銀行のような台帳係を置かずに、同じコインを二人に支払えないと主張できるのか。

答えの骨格は価格チャートでも「ブロックチェーン」というラベルでもない。Bitcoinが解いている問題は、デジタルデータがほとんど無料で複製できる世界で、二重支払い（double-spending）をどう止めるかである(*1)。信頼最小化とは「誰も信じなくてよい」ではなく、信じる対象を特定企業の内部帳簿から、公開された規則と検証可能なコストへ移すことである。

四つの役割は別物である

権限（署名） 状態変換（UTXO） コスト境界（PoW） 検証（フルノード）

Lesson: 「ブロックチェーン」一語では、権限・状態・コスト・検証が潰れてしまう。

1. N1 — 二重支払い問題：なぜ「ファイル送金」は現金にならないか

問題（前層の欠如）

物理の現金は、渡すと手元から減る。デジタルのファイルは、送っても手元のコピーが残りうる。オンラインで「コイン」を送るとき、受け手は「同じコインが別の人にも送られていないか」を確かめる必要がある。

古典的な解は、信頼できる第三者が残高を管理することである。Satoshi Nakamotoの提案は、この第三者を必須としない電子現金を求める(*1)。デジタル署名だけでは足りない——署名は「Aliceが承認した」ことは示せるが、「残高がまだその支払いを許す状態にある」ことは、グローバルな状態を見なければ示せない(*1)。

仕組み — 四点セット

  1. 権限の表現（→ N3 署名）
  2. 状態の表現（→ N4 UTXO）
  3. 履歴の合意（→ N6, N7 PoWと鎖選択）
  4. 検証の分散（→ N8 フルノード）

worked example — Aliceの二重支払い

AliceがコインCをBobに送るTx1を流す。Bobは受取と判断する。しかしAliceは同じCを入力にしたTx2をCarolへ同時に流す。第三者がいなければ、ネットワーク全体が共有する「Cはもう使われた」記録が要る。銀行がいなければ、全員が同じ順序の履歴を共有し、その履歴上でCは一度しか消費されない状態を作らなければならない。

次の限界

共有記録が要ると決まっても、「記録が改ざんされていない」ことを安く確認する道具がなくてはならない。ここで暗号ハッシュが必要になる。

2. N2 — ハッシュ：履歴を指紋に縛る

問題（前層の破綻）

履歴ファイルをそのまま配ると、大きく、一点の改変を見逃しやすい。P6の「改ざん不可能」も、まずは「改ざんを検出できる」という弱い主張から始まる。

仕組み

同じ入力なら同じ出力、入力を1ビット変えると出力が予測不能に変わる、逆算は実務上不可能——この性質の束が、後の「鎖」と「PoW」の糊になる(*2)(*1)。ブロックB₂のヘッダにはB₁のヘッダハッシュが含まれる。B₁の中身を1バイト変えると、B₂以降すべてのハッシュも連鎖的に無効になる。

worked example

取引集合の要約指紋が H1 だとする。受取人を差し替えると H2 になり、検証者は中身を読まなくても不一致を検出できる。ハッシュは改ざんを検出する。改ざんを不可能にするのは、後のPoWが作り直しに膨大なコストを課すからである。

次の限界

指紋は改変検出を与えるが、「誰がこのコインを動かしてよいのか」は与えない。

3. N3 — 公開鍵署名：中央名簿なしの所有

問題（前層の破綻）

Bitcoinは世界共通の氏名台帳を前提にしたくない。それでも「この残高を動かしてよいのは鍵の持ち主だけ」と言いたい。

仕組み

秘密鍵から公開鍵を導出し、ハッシュを取ってアドレス（bc1... など）として流通する。アドレスは「Alice」という名前ではなく、支払い条件のハッシュである(*3)。ネットワークは人物ではなく署名を検証する。

  • 鍵を持つ者が主権を持つ（第三者の許可なく使える）
  • 鍵を失う・盗まれると、プロトコル上の救済当局はいない（P4）

worked example — 取引所IOU vs 鍵主権

Aliceが取引所にBTCを預けている場合、画面の残高は内部帳簿のIOUである。取引所が破綻・ハッキング・出金停止をしたら、画面の数字は紙切れ同然。P4の裏返し——救済がない代わりに、第三者にも止められない。

次の限界

権限があっても、グローバルな「未使用状態」がなければ二重支払いが戻る。

4. N4 — UTXO取引：残高は口座ではなく未使用アウトプット

問題（前層の破綻）

口座残高モデルだと更新順序の合意に強い中央性が入りやすい。Bitcoinは未使用トランザクションアウトプット（UTXO）の集合を状態にする(*3)。各UTXOは過去の取引の出力スロットであり、一度消費されたら二度と使えない。

仕組み

  1. 既存UTXOを入力として参照し、消費する
  2. 新しいUTXOを出力として作成する
  3. 入力額の合計 ≥ 出力額の合計。差額は手数料
  4. 各入力に消費権限の証明（署名）を付ける

ウォレットの「残高 1.42 BTC」は、使えるUTXOの合計という便利な集計である。原子単位は口座行ではなくUTXOである。

UTXO Lab（N4）

Hard claim: 同じ UTXO を二つの取引で使おうとすると、二つ目は台帳状態と矛盾し拒否される。

入力UTXOを選び、Bobへの支払いとお釣りを配分して取引を構築する。同じ入力で第二の取引を試みると拒否される。

未使用 UTXO（Alice）

台帳 / mempool

未使用集合を表示

入力を選んで取引を構築してください。

Lesson: 残高はUTXOの集計。入力は一度消費すると未使用集合から消える。

静的: 消費済み入力を第二取引で再利用 → reject（台帳矛盾）。

worked example — お釣りと手数料

項目金額
入力0.8 BTC のUTXO（Alice署名）
出力1Bobへ 0.3 BTC
出力2Aliceへ 0.499 BTC（お釣り）
手数料0.001 BTC

Aliceが同じ0.8 BTC UTXOをCarol向けの第二取引でも使おうとすると、先に受理された方だけが有効で、後から来た方は二重支払いとして拒否される。

次の限界

有効取引を世界が一本ずつ高コストで「石に刻む」のは重い。取引を束ねて、包含を要約するブロックが要る。

5. N5 — ブロックとMerkle根：履歴を束ね、包含を要約する

問題（前層の破綻）

取引ごとに独立の高コスト証明を要求すると同期と検証が破綻する。巨大な取引袋だけをP2Pで配ると転送が重く、軽い包含証明も効かない。

仕組み — ブロックの二層構造

ブロックはコンパクトなヘッダ（約80バイト）と可変長の取引本体からなる(*2)。PoWが計算する対象は主にこの80バイトのヘッダである。取引はMerkle根を経由してヘッダに拘束される。

フィールド役割
prevBlockHash直前ブロックヘッダのハッシュ（鎖の糊）
merkleRoot当ブロック内全取引のMerkle根
bits / nonce難易度目標とPoW探索用カウンタ

worked example — 4取引のMerkle木

        Root = H(H(AB) ‖ H(CD))
       /                        \
  H(AB)                    H(CD)
   /    \                    /    \
 H(A)  H(B)              H(C)  H(D)

Bobは取引Bだけ知りたい。Merkle proof として {H(A), H(CD)} を受け取り、根を再計算してヘッダの merkleRoot と照合する。Bを差し替えると根が一致しない。

コインベース取引

各ブロックの先頭取引はコインベース取引。Genesisの coinbase には The Times 紙の見出しが刻まれている(*1)。

SPVとの境界（P6）

SPVクライアントはヘッダ鎖と Merkle proof だけで確認しようとするが、コンセンサスルール全体は検証しない(*1)。P6の「改ざん不可能」は自分が検証した範囲に限定される。

次の限界

誰でもヘッダを書いてよいと、矛盾するブロックが無限に増える。追記提案に費用を載せる。

6. N6 — Proof-of-Work：追記権に費用を課す

問題（前層の破綻）

提案が無料なら、攻撃者は矛盾する履歴を量産できる。オンラインでは新しい「身分」を作るコストがほぼゼロであり、票＝計算コストにしない限り多数決は偽身分に飲まれる。

仕組み — Hashcashの血統

Adam Backの Hashcashは、メール送信者にハッシュ探索を要求するDoS対策だった(*5)。探索は高く、検証は安いという非対称性である。Satoshi Nakamotoはこの発想を「次の正規履歴候補を提案する権利」に再配置した(*1)。

PoW Lab（N6）

Hard claim: 有効 PoW の発見は nonce 探索で高コスト、検証は1回のハッシュで即座に完了する。

簡略モデル: ハッシュ先頭が 000 で始まる nonce を探す。実 Bitcoin は SHA-256d とより厳しい target を使う。

マイナー（探索）

試行回数: 0

現在 nonce: 0

—

フルノード（検証）

検証時間: —

探索を開始してください。

Lesson: 提案の希少性を、検証可能な形で示せる。無料提案が履歴を汚染する理由がここにある。

静的: 先頭ゼロ条件を満たす nonce は多数試行が要り、検証は1回で足りる。

worked example

target が「先頭20ビットが0」程度だと、平均試行回数は指数関数的に増える。攻撃者が過去ブロックを書き換えようとすると、そのブロック以降のすべてのブロックについて、ネットワークが積み上げた作業量を単独で再実行しなければならない。6確認（P3）はこの再実行コストが実務上無視できるほど大きい、という経験則に基づく。

次の限界

遅延のあるネットワークでは、同じ高さに二つの有効ブロックが一時的に並ぶ（フォーク）。どれを伸ばすかの規則が要る。

7. N7 — 累積作業量：フォークのあいだで正規履歴を選ぶ

問題（前層の破綻）

同時出塊や伝播遅延で短い分岐は不可避である。確定は確率的・コスト的な概念になる。

仕組み

ノードは、コンセンサスルールを満たす鎖のうち、最も累積Proof-of-Workが大きいものを正規とする。論文の "longest chain" は実務上累積作業量（chainwork）として読むのが安全である(*1)(*2)。

Fork choice Lab（N7）

Hard claim: 累積作業量が大きい枝だけが正規履歴として残り、短い枝にだけ載った取引はなかったことになりうる。

高さ N で枝 A / B に分岐したあと、どちらを伸ばすか選ぶ。正規 tip は累積 work が大きい側。

枝 A

累積 work: 10
共有祖先上の取引

枝 B

累積 work: 10
TxB: Bにしかない支払い

Lesson: 確認数＝差し替えコスト。取引所の「反映済み」（P1）と自分の確認時計は別物。

静的: work(A)>work(B) なら tip=A で TxB は orphaned。

worked example — 再編成とP1

Alice→Bob の取引がY側のブロックにだけ入ったとする。Bobが商品を渡した後、X側の累積作業量が上回れば再編成が起き、Y側の取引は孤児（orphan）になる。P3（6確認）は攻撃者の再実行コストが実務上十分大きくなるまで待つ経験則である。

次の限界

マイナーが作ったから正しい、ではない。規則違反のブロックはPoWを満たしていても無効である。

8. N8 — P2P検証：フルノードが立法者ではなく審査官になる

問題（前層の破綻）

中央サーバがないなら、誰が無効データを止めるのか。マイナーが「報酬2倍のコインベース」を載せたブロックをPoW成功させても、ネットワークは従ってはならない。ルールはプロトコルにあり、マイナーは候補の提案者に過ぎない。

仕組み — フルノードの検証パイプライン

フルノードはピアから取引とブロックを受け取るたびに、ローカルのコンセンサスルールで検証する(*4)。拒否されたデータは自分のUTXOセットにも、relay経路にも乗らない。

Validation Lab（N8）

Hard claim: 不正取引/ブロックは PoW 風の表示があってもフルノード規則で拒否される。

無効ケースをトグルし、フルノードへ提出する。PoW表示は常に「OK（見かけ）」でも、規則違反なら reject。

攻撃者の表示: PoW: OK（見かけ） — 実際の検証とは独立

    ケースを選んで提出してください。

    Lesson: PoWの「仕事」は、ルール内のブロックにだけ意味を持つ。マイナー ≠ 立法者。

    静的: いずれかの規則違反 → reject（PoW表示は無関係）。

    worked example — 過大コインベースの拒否

    攻撃者がPoWを成功させたブロックに「コインベースで1000 BTC」を載せる。ハッシュは target を満たしている。しかしフルノードはコインベース報酬上限ルールで即拒否する。攻撃者の鎖は伸びない。

    次の限界

    検証と探索には資源が要る。なぜ誰かがエネルギーを払うのか——発行と手数料へ。

    9. N9 — 発行・手数料・難易度：セキュリティ予算と時間リズム

    仕組み

    ブロック報酬は補助金（block subsidy）と取引手数料の合計である(*2)。補助金は約210,000ブロック（約4年）ごとに半減する。P5：半減は供給スケジュールのイベントであり、そのままハッシュレート・価格・「安全性」を保証しない。

    P2（手数料の変動）：ブロック容量は有限である。混雑時、マイナーは手数料率（sat/vB）の高い取引を優先する。ユーザーが見ているのは「BTCの値段」だけでなくブロック空間の一時市場である。難易度調整は時間リズムを保つ(*6)。

    worked example — mempool入札

    混雑時、mempoolに10,000件が滞留し次のブロックには約3,000件しか入らない。Alice（5 sat/vB）は数ブロック待ち、Bob（50 sat/vB）は次ブロックで確認される。

    10. N10 — 限界と、Bitcoinが解かない問題

    領域限界の根拠
    価格2100万 capは供給上限であり、需要は市場（プロトコル外）
    完全匿名全取引は公開台帳。連鎖分析で追跡可能
    無限スループットオンチェーンは意図的に制約（~7 tx/s程度）
    鍵の社会的回復シード喪失はプロトコル外（P4）
    消費者保護チャージバック前提の決済法制とは設計思想が異なる
    エネルギーゼロPoWのコストは防衛の対価

    よくある誤解

    • 「ブロックチェーンと書けば改ざん不可能」→ 累積PoWと受理規則に条件付き（P6）
    • 「マイナーがすべてを決める」→ 無効ブロックはフルノードが拒否(*4)
    • 「取引所の残高＝自分のBitcoin」→ 鍵制御が本義（IOU）
    • 「半減期＝必ず稀少性バブル」→ 供給スケジュールの話（P5）
